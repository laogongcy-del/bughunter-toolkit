# 链式攻击思维

> 单个低危漏洞价值有限，组合起来才能造成真正的危害

## 核心观点

**单一漏洞 ≠ 高价值漏洞**

一个反射XSS可能只值低危，但如果能配合CSRF变成蠕虫，配合Cookie窃取变成账号接管，危害就完全不同了。

## 常见攻击链模式

### 1. 信息泄露 → 凭据获取 → 后台接管

```
信息泄露
  ├─ JS文件泄露API Key/Token
  ├─ 目录遍历读取配置文件
  ├─ 源码泄露（.git/.svn）
  └─ 第三方服务泄露（GitHub、Pastebin）
      ↓
凭据获取
  ├─ 泄露的API Key直接用于API调用
  ├─ 从配置文件中找到数据库密码
  ├─ Token通过JWT解码获取身份信息
      ↓
后台接管
  ├─ API Key有管理员权限
  ├─ 数据库密码直连读数据
  └─ Token伪造接管账号
```

**案例**：通过`.git`泄露获取源码 → 源码中找到数据库密码 → 连接数据库发现管理员Hash → 破解后登录后台 → 后台有文件上传功能 → Getshell

### 2. 文件上传 → 中间件 → 内网漫游

```
文件上传
  └─ 上传图片→服务端处理（如SSRF）
      ↓
SSRF到内网
  ├─ 扫描内网端口
  ├─ 访问内网服务（Redis、MySQL等）
  └─ 访问云元数据接口拿临时凭证
      ↓
横向移动
  ├─ 利用Redis未授权写SSH密钥
  ├─ 利用云凭证接管云服务
  └─ 通过MySQL读文件获取更多敏感信息
```

**案例**：头像上传功能 → 服务端用curl处理图片URL → 构造SSRF → 访问AWS元数据 → 获取IAM临时凭证 → 接管S3存储

### 3. XSS → Cookie窃取 → 账号接管

```
XSS（反射/存储）
  └─ 在目标域执行JS
      ↓
Cookie窃取
  ├─ document.cookie发送到攻击者服务器
  ├─ 窃取LocalStorage/SessionStorage
  └─ CSRF Token窃取
      ↓
账号接管
  ├─ 使用窃取的Cookie直接登录
  ├─ 利用CSRF执行敏感操作
  └─ 结合点击劫持提权
```

### 4. 子域名接管 → 同源绕过 → 数据窃取

```
子域名fuzz
  └─ 发现CNAME指向第三方但未注册
      ↓
域名接管
  ├─ 注册该域名（云服务/Heroku/GitHub Pages）
  ├─ 在该域名上托管恶意JS
      ↓
同源策略绕过 / 其他
  ├─ 通过同域存储窃取Cookie
  ├─ 通过父域Cookie作用域扩大攻击面
  └─ 结合host header注入
```

**经典案例**：子域名指向AWS S3 → Bucket已被删除 → 重新创建该Bucket → 在Bucket上托管文件 → 同域下可访问原站Cookie

### 5. API逻辑链

```
低权限API
  └─ 发现正常API返回数据中的敏感字段
      ↓
参数篡改
  ├─ 替换user_id遍历其他用户数据
  ├─ 替换role/group_id权限提升
  └─ 跳过状态参数
      ↓
横向/纵向越权
  ├─ 查看其他用户隐私数据
  ├─ 以管理员身份操作
  └─ 绕过付费/权限检查
```

## 如何发现攻击链

### 第一步：单点发现
在测试时记录所有"小问题"：
- 信息泄露（即使只是路径泄露）
- 功能点异常（接口返回了不该有的数据）
- 权限校验不一致（有些接口鉴权，有些没有）
- 第三方服务交互（上传、回调、webhook）
- 用户输入输出点（所有可能被利用的入口）

### 第二步：建立关联
把看似不相关的问题关联起来：
```
┌──── A: 信息泄露 ────┐
│  /api/users 未授权    │
│  返回用户手机号       │
└──────────────────────┘
          ↓ 关联
┌──── B: 功能点 ──────┐
│  重置密码需要手机号   │
│  验证码4位可爆破      │
└──────────────────────┘
          ↓ 攻击链
A获取手机号 → B用手机号爆破验证码 → 任意账号密码重置
```

### 第三步：由小到大
不要只看眼前的危害，问自己：
- 这个信息还能用在什么地方？
- 这个接口还能组合什么操作？
- 这个权限还能访问哪些资源？
- 如果我有A + B，能做什么？

## 攻击链的价值判断

| 链长度 | 价值 | 提交建议 |
|--------|------|---------|
| 单点漏洞 | 低-中危 | 直接提交或寻找组合 |
| 2步链 | 中危 | 可以提交，也可继续挖掘 |
| 3步+链 | 高危-严重 | 完整提交，这是高质量报告 |

## 实战建议

1. **记录所有发现** — 建个笔记记下所有异常，哪怕看似无用的
2. **画攻击链路图** — 用简单的箭头图画关联
3. **别止步于第一步** — 每个发现都问"然后呢？"
4. **跳出技术思维** — 想想业务流中的异常
5. **关注数据流向** — 数据从哪里来，经过什么处理，存到哪里
6. **关注第三方交互** — SSRF、回调、Webhook 是攻击链的黄金节点

---

> ⚠️ **合规声明**: 本方法论仅用于已获得明确授权的安全测试。攻击链测试应在受控环境中进行，不得在未授权目标上执行完整的链式攻击。
