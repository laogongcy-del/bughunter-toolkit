# 业务逻辑漏洞挖掘

> 业务逻辑漏洞是"功能设计缺陷"，技术手段无法完全防御

## 什么是业务逻辑漏洞

业务逻辑漏洞（Business Logic Vulnerability）不是传统技术漏洞（如SQL注入、XSS），而是应用程序业务流中的设计缺陷。通常自动扫描器无法发现，需要手工测试。

**典型特征**：
- 代码没有漏洞，但流程设计有缺陷
- 正常功能被恶意利用
- 业务规则可以被绕过

## 1. 支付逻辑漏洞

### 1.1 价格篡改
```
HTTP请求拦截 → 修改金额参数
├─ 修改JSON中的 price/amount/total
├─ 修改负数金额
├─ 修改金额为0.01或0
├─ 修改数量为负数（退款变入账）
└─ 修改优惠券抵扣额超过订单金额
```

**检测方法**：
- 下单时抓包，搜索金额相关字段（price, amount, total, fee, cost）
- 逐个修改字段值看响应
- 关注浮点数精度问题（0.01 变 0.0099999）

### 1.2 优惠券滥用
```
优惠券/促销码相关漏洞
├─ 同一张优惠券反复使用（并发/重放）
├─ 优惠券叠加使用（应该互斥的券同时使用）
├─ 使用不属于自己的优惠券（越权）
├─ 修改优惠券面额
├─ 绕过优惠券使用条件（满XX减XX，不满额）
└─ 新客优惠券被老用户使用
```

### 1.3 支付跳过快
```
支付流程
├─ 下单 → 支付确认 → 支付成功（正常流程）
├─ 下单 → 直接请求"支付成功"API → 跳过实际支付
├─ 下单 → 取消支付 → 但订单状态仍被更新为已支付
└─ 减少支付步骤（多步支付只走最后一步）
```

**检测方法**：
- 走一遍完整支付流程，记录所有请求
- 尝试跳过中间步骤直接请求最后一步
- 在支付确认页面回退/刷新看状态

## 2. 注册/登录逻辑漏洞

### 2.1 注册绕过
```
注册流程中的漏洞
├─ 无需邮箱/手机验证即可注册
├─ 邮箱/手机验证码可爆破
├─ 验证码返回在响应中
├─ 使用已注册邮箱但覆盖密码
├─ 注册时修改role参数提权
└─ OAuth注册绕过（直接POST创建用户）
```

### 2.2 身份验证绕过
```
登录绕过
├─ 直接访问登录后页面（未授权访问）
├─ 修改响应状态码（302→200）
├─ Cookie伪造
├─ JWT修改（alg:none）
├─ 修改请求方法绕过登录检查
└─ SQL注入登录（万能密码）
```

### 2.3 密码重置缺陷
```
密码重置逻辑
├─ Token可预测（时间戳/顺序数字/短字符串）
├─ Token在响应中返回
├─ Token不校验绑定用户（篡改用户ID）
├─ 验证码可爆破（4位/6位无限制）
├─ 新旧密码相同时仍可重置
└─ 重置后自动登录但返回旧会话
```

**经典案例**：
1. 请求密码重置 → 拦截邮件中的链接 → Token是base64(user:id:timestamp) → 解码后伪造任意用户Token
2. 密码重置 → 验证码6位数字 → POST到API无速率限制 → 1万次暴力破解 → 任意账号重置

## 3. 越权操作（IDOR）

详见 [技术手册 - 越权检测](../techniques/02-idor.md)

### 3.1 水平越权
- 修改用户ID查看他人数据
- 修改订单ID查看他人订单
- 修改邮箱/手机号获取他人信息

### 3.2 垂直越权
- 普通用户执行管理操作
- 修改角色/权限字段
- 访问管理接口

### 3.3 特殊技巧
- 参数位置切换（GET → POST body）
- 数据格式切换（JSON → XML → form）
- 添加额外字段
- 删除鉴权字段

## 4. 多步操作跳过

### 4.1 流程绕过
```
应用有N步操作
├─ 步骤1: 填写信息
├─ 步骤2: 确认信息
├─ 步骤3: 审核
├─ 步骤4: 完成
跳过检测: 直接请求步骤4的接口
```

**典型场景**：
- 用户注册→邮箱验证→完善资料 → 跳过邮箱验证
- 添加商品→填写详情→提交审核 → 跳过审核直接发布
- 上传文档→支付→下载结果 → 跳过支付

### 4.2 状态篡改
- 修改订单状态（pending→completed/shipped/cancelled）
- 跳过状态机中的中间状态
- 从状态A直接跳转到状态C

## 5. 速率限制缺陷

### 5.1 无限制操作
```
无速率限制的风险点
├─ 登录尝试（爆破密码）
├─ 验证码发送（短信轰炸）
├─ 验证码校验（暴力破解）
├─ API调用（数据爬取）
└─ 评论/发帖（垃圾信息）
```

### 5.2 速率限制绕过
```
绕过方法
├─ X-Forwarded-For头部轮换
├─ Cookie/Token重置
├─ 参数变化（加随机参数）
├─ HTTP/1.1 keep-alive绕过
├─ 使用WebSocket代替HTTP
└─ 使用GraphQL batching
```

## 6. 竞争条件（Race Condition）

详见 [技术手册 - 条件竞争](../techniques/06-race-condition.md)

### 6.1 常见场景
- 优惠券多次使用
- 余额多次提现
- 库存超卖
- 评论/点赞重复

### 6.2 简单测试方法
```python
import threading
import requests

def race_test(url, cookies, times=50):
    results = []

    def req():
        r = requests.get(url, cookies=cookies)
        results.append(r.status_code)

    threads = [threading.Thread(target=req) for _ in range(times)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    # 观察是否有多于预期的成功响应
    success = results.count(200)
    print(f"成功: {success}/{times}")
    if success > 1:
        print("可能存在竞争条件漏洞")
```

## 7. Mass Assignment（批量赋值）

### 7.1 原理
应用自动将请求参数绑定到对象属性，未过滤敏感字段。

### 7.2 测试方法
```http
# 正常请求
POST /api/user/update
{"name": "新名字"}

# 添加额外字段
POST /api/user/update
{"name": "新名字", "role": "admin", "isAdmin": true}

# 如果成功，就是Mass Assignment
```

### 7.3 常见敏感字段
```
role, isAdmin, is_active, verified, email_verified
is_premium, vip_level, points, balance, credit
is_paid, status, is_deleted, permissions
```

## 8. 搜索/数据泄露

### 8.1 搜索功能
```
搜索相关漏洞
├─ 搜索未授权数据（搜到他人订单）
├─ SQL注入搜索框
├─ 搜索参数泄露
├─ 搜索结果包含敏感信息
└─ 搜索建议泄露（自动补全）
```

### 8.2 遍历/枚举
```
ID遍历
├─ /order/1001 → /order/1002 → ...
├─ /user/profile?uid=100 → 101 → 102
└─ Base64/UUID递增也可遍历
```
