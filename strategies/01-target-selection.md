## 1. 新手选目标三大原则

### 1.1 选能注册的 —— 有业务逻辑才能测越权

**核心理念**：漏洞的本质是业务逻辑的缺陷。如果一个网站你连账号都没有，你只能测测前端页面、扫扫目录，攻击面极其有限。而一旦你有了一个普通用户账号，你能测试的漏洞类型立刻翻倍：

- **越权漏洞**（IDOR）：普通用户能不能访问管理员页面？能不能看别人的订单？
- **逻辑漏洞**：能不能越权支付、绕过限额、修改他人资料？
- **水平越权**：能不能用A账号看到B账号的数据？

**实战案例**：
> 某教育平台，注册后获得学生角色。通过修改URL中的 `user_id=1001` 为 `user_id=1002`，直接查看到其他学生的姓名、手机号、身份证号。这就是一个典型的高危水平越权漏洞。

**反例**：
> 某纯展示类政府网站，没有登录功能，只有新闻公告和办事指南。攻击面只有一个搜索框和一个留言板，能挖的只有反射XSS（补天不收）和SQL注入（概率极低），投入产出比极差。

**实操建议**：
1. 在目标清单中优先选择**有注册入口**的站点
2. 注册时注意：是否需要手机验证？是否需要邮箱验证？是否需要实名认证？
3. 如果一个目标注册需要邀请码/审核通过，说明攻击面较窄但价值可能更高（内部系统）
4. 优先选择**注册即用**的目标，不要选需要管理员审核账号的

### 1.2 选中小厂/教育/政务 —— WAF弱，容易出洞

**为什么中小厂容易出洞？**

| 维度 | 大厂 | 中小厂 | 教育/政务 |
|------|------|--------|-----------|
| 安全团队 | 专门的蓝队、红队 | 兼职/外包 | 基本没有 |
| WAF | 自研WAF + 云WAF | 简单WAF或无 | 无/过时WAF |
| 代码审计 | 强制代码安全审查 | 无 | 无 |
| 漏洞奖励 | 高（数万起） | 中（数百到数千） | 低（荣誉为主） |
| 漏洞发现难度 | 极高 | 低 | 极低 |

**教育行业为什么是金矿**：

- 教育系统（尤其是高校教务系统）通常由**外包公司**开发，代码质量堪忧
- 上线后很少做安全维护
- 学生数据（身份证、银行卡号等）敏感程度高，漏洞价值大
- 补天收录教育行业的漏洞

**真实案例**：
> 某大学选课系统，登录后发现查询成绩接口存在 SQL 注入，直接拿到教务系统数据库，包含全校师生的姓名、身份证号、手机号、家庭住址。这个漏洞在补天被认定为**高危**。

**政务系统**：

- 政府网站现在普遍上云，但安全配置往往不到位
- 政务系统接口多（办事、查询、缴费），攻击面大
- 敏感数据（身份证、不动产信息、企业信息）价值高
- 部分政务系统使用老旧框架（Struts2、ThinkPHP低版本等）

**注意**：政务系统需要特别注意合规性，确保在授权范围内测试。补天平台上有明确的政务目标授权列表，不要越界。

### 1.3 选新上线的 —— 刚上线漏洞多

**为什么新上线系统漏洞多？**

1. **开发阶段安全测试不充分**：项目赶工期，安全测试被压缩
2. **配置遗留**：默认密码、调试接口未关闭、测试账号未删除
3. **第三方组件版本旧**：上线时可能用了当时的最新版，但从开发到上线可能有数月时间差
4. **CDN/WAF配置不完善**：刚上线时安全防护往往后知后觉

**如何发现新上线的目标**：

1. **补天平台**：关注「新增目标」列表，每周查看
2. **子域名监控**：使用自己搭建的子域名监控系统（如 Subfinder + HTTPX + 定时任务），自动发现新子域名
3. **网站备案信息**：查询近30天备案的新域名
4. **FOFA/SHODAN 新资产**：定期搜索特定关键词，按时间排序
5. **招聘信息**：看目标公司招聘安全工程师，说明可能正在加强安全，趁还没加强之前赶紧挖

**新系统常见漏洞**：

```
1. 管理员默认密码：admin/admin123, admin/123456
2. Swagger API 文档未关闭：/swagger-ui.html, /api/doc
3. 测试账号未删除：test/test123, demo/demo123
4. 调试接口开放：/debug, /api/test, /druid
5. 敏感信息泄露：/.git, /.env, /backup, /WEB-INF/web.xml
6. 未授权访问：新系统权限控制往往有遗漏
```

**实战技巧**：
> 发现一个刚上线的电商平台，登录后查看API返回的JSON中发现 `"is_admin": false`。修改为 `"is_admin": true` 后获得管理员权限。这就是新系统常见的「前端控制权限、后端未校验」的经典漏洞。

## 2. 补天平台目标筛选

### 2.1 如何看补天目标列表

补天平台的目标列表是白帽子最重要的资源。以下是如何高效利用：

**目标列表核心信息**：

| 字段 | 含义 | 筛选建议 |
|------|------|----------|
| 目标名称 | 企业/系统名称 | 关注你有了解的行业 |
| 目标类型 | 域名/IP/APP | 优先域名，其次APP |
| 测试范围 | 具体域名/路径 | 越大越好 |
| 赏金范围 | 漏洞奖励金额 | 不要只看钱，要看难度 |
| 授权状态 | 已认证/未认证 | 选已认证的 |
| 参与人数 | 正在测试的人数 | 人太多的避开 |
| 提交漏洞数 | 已有漏洞提交统计 | 漏洞太多可能难挖了 |

**高效筛选方法**：

1. **排序方式**：按「最新发布」排序，抢新目标
2. **筛选条件**：选择「企业认证」「有保底奖励」
3. **排除条件**：排除「仅限邀请」「企业内测」
4. **重点关注**：SaaS平台、电商系统、OA系统、ERP系统

### 2.2 哪些标签代表有搞头

**高价值标签**（优先选择）：

| 标签 | 原因 |
|------|------|
| SaaS平台 | 多租户架构，越权漏洞多发 |
| 电商系统 | 业务复杂，支付/订单/优惠券都有搞头 |
| OA系统 | 权限体系复杂，水平越权常见 |
| ERP系统 | 数据量大，SQL注入价值高 |
| 云平台 | 配置错误导致的大量数据泄露 |
| 金融系统 | 高危漏洞的高价值数据（但WAF普遍较强） |
| 医疗系统 | 患者数据敏感度高（HIPAA合规） |
| 教育平台 | 学生信息价值高，且普遍安全薄弱 |
| 政务平台 | 数据敏感，安全措施相对薄弱 |
| 物联网平台 | 鉴权漏洞、默认密码常见 |

**中价值标签**：

| 标签 | 原因 |
|------|------|
| 内容管理系统(CMS) | 模板注入、文件上传漏洞常见 |
| 在线考试系统 | 逻辑漏洞（替考、查答案） |
| 视频会议平台 | 会议鉴权绕过 |
| 企业官网（带后台） | 后台弱密码 |
| 酒店预订系统 | 越权查看订单 |
| 物流系统 | 越权查看客户信息 |

### 2.3 哪些标签代表没搞头

**低价值/无价值标签**（减少投入）：

| 标签 | 原因 |
|------|------|
| 纯展示站 | 没登录、没交互、纯静态HTML |
| 企业官网（无后台） | 攻击面只有留言板和搜索 |
| 个人博客 | 攻击面极小 |
| 论坛/社区（老旧） | 如果运营多年还没被黑，说明已经没漏洞了 |
| 镜像站/缓存站 | 无实际业务 |
| 域名停放页面 | 无内容 |
| 仅APP（无Web端） | APP逆向成本高，新手不推荐 |
| 只给一个IP的 | 攻击面极窄 |

**注意**："没搞头"不等于"绝对没漏洞"，只是投入产出比太低。一个纯展示站你花两周可能找到1个反射XSS（补天还不一定收），而一个SaaS平台你可能花两天找到3个高危漏洞。

### 2.4 目标资产范围确认方法

**为什么要确认资产范围**：

在补天提交漏洞，最尴尬的情况就是：你发现了一个漏洞，结果审核告诉你「不在测试范围内」。所以**测试前必须确认清楚目标范围**。

**确认方法**：

1. **查看补天目标详情页**：
   - 明确列出的域名列表
   - 明确列出的IP段
   - 附属资产列表

2. **根域名推导**：
   - 如果目标范围是 `example.com`，通常意味着所有子域名都在范围内
   - 但如果目标范围是 `www.example.com`，则其他子域名可能不在范围内
   - 不确定时先问补天客服

3. **如何向补天申请扩大范围**：
   - 发现目标子域名下的资产不在范围内，可以提交申请
   - 提供资产属于该企业的证据（如备案信息、页面底部版权信息）
   - 说明该资产存在的安全隐患

4. **常见范围陷阱**：
   ```
   ❌ 只给了主域名，但实际业务在子域名
   ❌ 只给了生产域名，但测试环境属于同一公司
   ❌ 给出的IP段包含了很多无关服务
   ✅ 正确的做法：列出所有相干域名 + 资产说明
   ```

## 3. 目标难度分级

### 3.1 ⭐ 简单：教育行业、中小企业官网、政务系统

**特点**：
- WAF较弱或无WAF
- 使用开源CMS或过时框架
- 无专职安全人员
- 开发外包，代码质量参差不齐

**预期投入产出**：
- 投入时间：1-3天
- 预期漏洞：SQL注入、越权、逻辑漏洞、XSS
- 预期数量：3-8个
- 补天评分：中危-高危

**推荐的测试思路**：

```
Day 1: 信息收集（子域名、端口、技术栈） → 注册账号 → 浏览功能
Day 2: 抓包分析请求 → 测试越权（改ID、改参数） → 测试SQL注入（' " 等）
Day 3: 深度测试文件上传、逻辑漏洞、未授权访问
```

**典型案例**：
> 某高校教务系统（ThinkPHP框架），通过注册学生账号，发现查询成绩接口存在SQL注入，直接拿到数据库权限。补天评分：**高危**，奖金2000元。

### 3.2 ⭐⭐ 中等：电商平台、SaaS系统、医疗系统

**特点**：
- 有基础WAF防护（云WAF或开源WAF）
- 业务逻辑复杂，攻击面大
- 可能有兼职安全人员
- 使用微服务架构，接口多

**预期投入产出**：
- 投入时间：3-7天
- 预期漏洞：越权、逻辑漏洞、存储XSS
- 预期数量：2-5个
- 补天评分：中危-高危

**推荐的测试思路**：

```
Day 1-2: 深入信息收集（JS分析、API文档、S3桶）
Day 3-4: 注册多账号 → 水平越权测试 → 垂直越权测试  
Day 5-6: 逻辑漏洞（优惠券盗用、支付绕过、积分篡改）
Day 7: 总结整理，编写漏洞报告
```

**WAF绕过技巧**：

遇到WAF不要慌，以下方法可以尝试绕过：
```
1. 大小写混合：SeLeCt * FrOm
2. 注释干扰：SEL/**/ECT
3. URL编码：%27 代替 '
4. 双编码：%2527
5. HTTP参数污染：id=1&id=2&id=3
6. 分块传输：Chunked Transfer Encoding
7. 换行/制表符混淆
```

**典型案例**：
> 某电商SaaS平台，注册商户账号后，通过修改API请求中的 `store_id` 参数，成功查看其他商户的订单数据，包含客户姓名、电话、地址。补天评分：**高危**，奖金5000元。

### 3.3 ⭐⭐⭐ 困难：大厂（阿里/腾讯/字节等）

**特点**：
- 拥有专业安全团队
- 自研WAF + 云WAF多层防护
- 强制安全开发生命周期（SDL）
- 所有代码经过安全审计
- 攻防对抗经验丰富

**预期投入产出**：
- 投入时间：1-4周
- 预期漏洞：逻辑漏洞、越权（需要深度理解业务）
- 预期数量：0-2个
- 补天评分：中危
- 奖金高（数千到数万）

**推荐的测试思路**：

```
Week 1: 深入理解业务逻辑（读帮助文档、用所有功能）
Week 2: 代码审计（如果能拿到前端源码）
Week 3: 业务逻辑挖掘（特定场景、边界条件）
Week 4: 组合漏洞测试（CSRF+越权、XSS+CSRF等）
```

**为什么大厂难挖**：
1. 常见漏洞类型（SQL注入、XSS）在开发阶段已被自动化工具发现修复
2. WAF会拦截大多数探测请求
3. 权限控制体系完善，简单的IDOR很少见
4. 安全响应速度快，漏洞生命周期短

**大厂漏洞挖掘技巧**：

```
1. 关注新业务/新产品（刚上线时防护最弱）
2. 关注边缘业务（非核心系统安全投入少）
3. 关注第三方集成（大厂用SaaS服务可能是薄弱环节）
4. 关注配置漏洞（OSS存储桶、Git泄露）
5. 关注逻辑漏洞（大厂的业务逻辑极其复杂，容易出漏）
```

**典型案例**：
> 某大厂新上线的微服务接口，在特定参数组合下出现越权查询用户详细信息（姓名、手机号、身份证号）。由于是新业务，权限校验有遗漏。补天评分：**高危**，奖金30000元。

### 3.4 ⭐⭐⭐⭐ 地狱：有专门安全团队的大厂

**特点**：
- 顶级的AI+人工+WAF多层防护
- 安全团队人员充足且经验丰富
- 有红蓝对抗机制
- 漏洞奖励高但是极难发现

**预期投入产出**：
- 投入时间：无上限
- 预期漏洞：极少数逻辑漏洞或0day
- 预期数量：不一定有
- 补天评分：高危-严重

**适合人群**：
- 大厂挖洞经验丰富的老手
- 有代码审计能力的安全研究员
- 时间充裕、不需要短期回报的人

**对新手建议**：
- 不要一上来就挑战大厂，容易打击信心
- 先从简单目标积累经验，1-2年后尝试大厂
- 参与大厂SRC的"新手任务"（有些大厂有针对新人的低难度目标）

## 4. 如何快速判断目标有没有搞头

### 4.1 首页看技术栈

**快速判断技术栈的方法**：

```bash
# 查看响应头
curl -sI https://target.com | grep -iE "server|x-powered|powered"

# 查看页面框架特征
curl -s https://target.com | grep -iE "thinkphp|laravel|wordpress|drupal|jquery|react|vue|angular"

# 查看是否有Web应用防火墙
curl -sI https://target.com | grep -iE "waf|cloudflare|akamai|incapsula|safedog|yundun|aliyun|qcloud"
```

**不同技术栈的"有搞头"指数**：

| 技术栈 | 有搞头指数 | 说明 |
|--------|-----------|------|
| ThinkPHP (≤5.x) | ★★★★★ | 已知RCE漏洞，框架漏洞多 |
| WordPress (旧版) | ★★★★☆ | 插件漏洞多 |
| Laravel | ★★★☆☆ | 相对安全，但配置错误常见 |
| Spring Boot | ★★★☆☆ | Actuator端点泄露常见 |
| ASP.NET | ★★☆☆☆ | 相对安全 |
| 纯静态HTML | ★☆☆☆☆ | 基本没搞头 |
| React SPA | ★★☆☆☆ | 前端逻辑可以在JS中找到 |
| 自己开发的框架 | ★★★★☆ | 没有经过广泛安全测试 |

**快速判断标准**：
- 使用了知名CMS且有版本号：可搜索该版本已知漏洞
- 没有版本号：尝试常见后台路径、默认文件
- 纯静态页面：大概率没搞头，除非有动态API
- 有API接口文档（Swagger）：宝藏

### 4.2 看有没有注册/登录

**登录功能的价值**：

| 注册/登录类型 | 价值评估 | 可测试的漏洞 |
|--------------|---------|-------------|
| 开放注册 | ★★★★★ | 越权、逻辑漏洞、信息泄露 |
| 手机/邮箱注册 | ★★★★☆ | 同上，但需要手机/邮箱资源 |
| 第三方登录 | ★★★☆☆ | OAuth配置问题 |
| 仅企业邮箱注册 | ★★☆☆☆ | 攻击面窄，价值高（内部系统） |
| 审核制注册 | ★☆☆☆☆ | 可以先分析的JS中找API |
| 没有注册 | ★☆☆☆☆ | 攻击面极其有限 |

**没有注册登录怎么办**：

1. 查看是否有**未授权访问的API**
2. 是否有**弱密码测试账号**（test/test123等）
3. 是否可以通过**默认密码**登录（admin/admin123）
4. 是否有**忘记密码功能**可以枚举/越权
5. 是否有**访客模式**可以用

### 4.3 看有没有API交互

**为什么要看API交互**：

现代Web应用大部分是前后端分离的，前端代码中包含了API的调用方式。通过分析API，你可以找到：

1. 所有可调用的接口
2. 接口的参数结构
3. 返回的数据格式
4. 接口之间的关联关系
5. 权限控制的实现方式

**如何抓包分析API**：

```bash
# 浏览器工具
1. 打开Chrome DevTools → Network 标签
2. 勾选 "Preserve log"
3. 浏览各个功能，观察所有请求
4. 重点看 XHR/Fetch 请求

# 使用Burp Suite
1. 设置浏览器代理到 Burp Suite (127.0.0.1:8080)
2. 在Proxy -> HTTP History 中查看所有请求
3. 右键 -> "Send to Repeater" 可以重放和修改请求
4. 使用 Intruder 进行自动化测试

# 使用命令行工具
curl -s https://api.target.com/endpoint -H "Authorization: Bearer token" | jq .
```

**API交互中的危险信号**：

```
1. API参数中包含 user_id、role_id、is_admin 等
   → 大概率存在越权漏洞，试试修改这些参数值

2. API返回的数据中包含其他用户的数据
   → 可能存在越权漏洞

3. API没有认证/鉴权头
   → 可能存在未授权访问

4. API参数中有JSON/XML数据
   → 可能存在注入漏洞

5. API有分页参数（page、limit、offset）
   → 可能存在信息泄露（无限翻页获取所有用户数据）
```

**API安全测试清单**：

```
□ 查看API文档是否开放（/swagger, /api/doc, /v2/api-docs）
□ 测试未认证的API调用
□ 测试越权API调用（修改user_id、tenant_id等）
□ 测试API参数注入（SQL注入、NoSQL注入、命令注入）
□ 测试API参数校验（整数溢出、参数类型混淆）
□ 测试API速率限制（爆破、枚举）
□ 测试API IDOR（水平越权）
□ 测试API批量操作（一次性查询大量数据）
□ 测试API GraphQL接口（introspection查询）
```

### 4.4 WAF检测

**为什么要检测WAF**：
WAF的存在直接影响漏洞挖掘的难度。如果目标有强WAF，你可能花大量时间绕WAF而没有结果。提前检测WAF可以帮你合理分配时间。

**快速检测WAF的方法**：

```bash
# 方法1：发送恶意请求看是否被封
curl -s "https://target.com/?id=1%27%20OR%20%271%27%3D%271"
# 如果返回 403/406/中间件错误页面等，说明有WAF

# 方法2：查看响应头
curl -sI "https://target.com" | grep -i "waf\|cf-ray\|x-sucuri-id\|x-powered-by\x-server\|server"

# 方法3：使用wafw00f工具
wafw00f https://target.com

# 方法4：发送非HTTP请求
curl -s --http1.0 "https://target.com" -H "User-Agent: () { :; }; /bin/bash -c 'id'"
# 检测Shellshock漏洞的同时也可以看WAF反应
```

**常见WAF特征**：

| WAF类型 | 特征响应头/Cookie | 封禁特征 |
|---------|-------------------|---------|
| Cloudflare | cf-ray、__cfduid | 返回Cloudflare 403页面 |
| 阿里云WAF | acw_tc | 返回阿里云WAF拦截页面 |
| 腾讯云WAF | tsec | 返回腾讯云WAF拦截页面 |
| 安全狗 (Safedog) | safedog-flow-item | 返回安全狗拦截页面 |
| 云锁 (Yunsuo) | yunsuo_session | 返回云锁拦截页面 |
| 360网站卫士 | wzws_session | 返回360拦截页面 |
| 百度云加速 | yunjiasu-nginx | 返回百度云加速拦截页面 |
| AWS WAF | x-amzn-RequestId | 返回403 Forbidden |
| ModSecurity | 无特定头 | 返回406 Not Acceptable |
| Naxsi | 无特定头 | 返回特殊错误页面 |

**WAF强度分级**：

```
无WAF: ★☆☆☆☆ 直接测试，几乎无限制
基础WAF: ★★★☆☆ 可以用常见技巧绕过
云WAF: ★★★★☆ 需要比较高级的绕过技术
自研WAF: ★★★★★ 需要特定绕过方法（通常有内部逻辑漏洞）
AI WAF: ★★★★★ 行为检测，传统绕过方法失效
```

**发现WAF后的决策**：

```
1. 弱WAF: 尝试绕过，花1-2天时间
2. 中等WAF: 有选择的绕过，避开WAF检测敏感参数
3. 强WAF: 放弃基于参数的注入类漏洞，转为挖掘逻辑漏洞和越权漏洞
4. 超强WAF（AI WAF）: 放弃传统的扫描方式，转为手动分析业务逻辑
```

## 5. 时间管理

### 5.1 一个新目标最多花多少时间做Recon

**Recon（信息收集）的时间分配**：

```
Phase 1 - 快速评估（30分钟）
├── 打开网站首页（2分钟）
├── 查看网站技术栈（5分钟）
├── 注册/登录（5分钟）
├── 简单浏览功能（10分钟）
├── WAF检测（3分钟）
└── 判断是否有搞头（5分钟）

Phase 2 - 基础Recon（2小时）
├── 子域名枚举（subfinder, amass）（30分钟）
├── 端口扫描（naabu, nmap）（30分钟）
├── 目录爆破（ffuf, gobuster）（30分钟）
├── JS分析（JSFinder, LinkFinder）（20分钟）
└── 技术栈指纹识别（10分钟）

Phase 3 - 深度Recon（1天）
├── 全端口扫描（30分钟）
├── 爬虫抓取所有链接（30分钟）
├── API分析（Burp Suite）（2小时）
├── GIT泄露检测（3小时）
├── S3存储桶检测（30分钟）
├── 子域名接管检测（30分钟）
└── Google Hacking（30分钟）

Phase 4 - 持续Recon（每天30分钟）
├── 监控新子域名
├── 监控新端口
├── 监控新端点
└── 监控JS文件变化
```

**不同难度目标的Recon时间分配**：

| 目标难度 | Phase 1 | Phase 2 | Phase 3 | 总时间 |
|---------|---------|---------|---------|--------|
| ⭐ 简单 | 30分钟 | 2小时 | 可选 | 2.5小时 |
| ⭐⭐ 中等 | 30分钟 | 2小时 | 1天 | 1.5天 |
| ⭐⭐⭐ 困难 | 30分钟 | 2小时 | 3天+ | 3天+ |
| ⭐⭐⭐⭐ 地狱 | 30分钟 | 2小时 | 持续监控 | 持续 |

### 5.2 什么时候该放弃转下一个

**放弃的黄金法则**：如果一个目标你投入了**预计时间的2倍**还没有产出，就应该考虑放弃或暂停。

**具体放弃时机**：

```
简单目标（预计1-3天）：
- 第1天：Recon + 基础测试
- 第2天：深度测试
- 第3天：如果没有发现任何有价值的漏洞 → 放弃

中等目标（预计3-7天）：
- 前3天：Recon + 基础测试
- 第4-5天：深度测试
- 第6-7天：如果只发现1个低危 → 考虑放弃

困难目标（预计1-4周）：
- 第1周：深度Recon + 业务理解
- 第2-3周：系统化测试
- 第4周：如果没有任何发现 → 暂停，过段时间再回来看
```

**应该放弃的信号**：

```
1. ✅ 已经完全理解了业务逻辑但没找到突破口
2. ✅ 所有常见漏洞类型都测试过了没结果
3. ✅ WAF太强，无法绕过
4. ✅ 目标根本没有注册/登录功能
5. ✅ 目标使用了定制化极强的框架且安全成熟度高
6. ❌ 只是懒了不想测（不要因为这个原因放弃）
7. ❌ 因为被WAF封了几次就放弃（这是正常的）
8. ❌ 因为看不懂业务逻辑就放弃（应该先学习）
```

**放弃不代表永远放弃**：
- 把目标加入监控列表，定期重新检查
- 新功能上线可能是新的攻击面
- 过段时间目标可能换技术栈或WAF

### 5.3 多个目标并行策略

**为什么并行更高效**：

1. **等待时间利用**：扫描器在跑A目标的子域名时，你可以手动测试B目标
2. **心情调节**：在一个目标上卡住了，换个目标换个思路
3. **收益最大化**：不同目标的产出周期不同，并行可以保证稳定产出

**推荐的并行策略**：

```
初级（2个目标并行）：
├── 目标A：主动手动测试（正在深度挖掘）
├── 目标B：自动化Recon + 被动收集（正在信息收集阶段）

中级（3-4个目标并行）：
├── 目标A：深度测试（投入60%时间）
├── 目标B：中等测试（投入30%时间）
├── 目标C：Recon阶段（投入10%时间，自动化进行）

高级（5个以上目标并行）：
├── 目标A：写得差不多了，准备提交（投入40%时间）
├── 目标B：深度测试（投入30%时间）
├── 目标C：中等测试（投入20%时间）
├── 目标D-E：Recon阶段（投入10%时间，自动化进行）
```

**时间分配示例**：

```
早上（9:00-10:00）：
  检查昨晚的扫描结果（子域名、端口、目录爆破）
  筛选有价值的信息，记录到笔记

上午（10:00-12:00）：
  深度测试当前主力目标（手工测试+Burp Suite）
  记录发现的漏洞

午休（12:00-13:30）：
  放松，让脑子休息

下午（13:30-17:00）：
  继续深度测试主力目标
  发现WAF太强时，切换逻辑漏洞测试

傍晚（17:00-18:00）：
  查看其他目标的自动化Recon结果
  发现有新子域名的，快速测试一下

晚上（20:00-22:00）：
  写漏洞报告，整理截图，提交补天
  研究新的攻击技术
```

**并行管理的工具推荐**：

```
任务管理: Notion / Obsidian / Trello
扫描监控: 自己搭建的 Telegram Bot / Discord Bot
笔记整理: Obsidian 本地笔记（推荐）+ 截图工具
漏洞报告: 本地Markdown + 截图
进度追踪: 简单的表格 Excel / 在线表格
```

### 6.2 目标优先级矩阵

```
                    攻击面大
                       │
     ┌─────────────────┼─────────────────┐
     │                 │                 │
     │  第二优先       │  第一优先       │
     │                 │                 │
  WAF │ 大厂边缘业务    │ 教育/政务/      │
  强  │ SaaS平台(成熟)  │ SaaS平台(新)    │
     │                 │ 中小企业        │
     │                 │                 │
     ├─────────────────┼─────────────────┤
     │                 │                 │
     │  放弃/暂缓      │  第三优先       │
     │                 │                 │
  WAF │ 大厂核心业务    │ 中小企业官网    │
  弱  │ (安全团队强)    │ (无WAF但功能少) │
     │                 │                 │
     └─────────────────┼─────────────────┘
                       │
                    攻击面小
```

**优先级解释**：

1. **第一优先**（右上）：
   - 攻击面大 + WAF弱
   - 教育行业、政务系统、新上线的SaaS、中小电商
   - 投入产出比最高，适合主力投入

2. **第二优先**（左上）：
   - 攻击面大 + WAF强
   - 大厂边缘业务、成熟的SaaS平台
   - 需要绕过WAF能力，适合有经验的白帽子

3. **第三优先**（右下）：
   - 攻击面小 + WAF弱
   - 中小企业官网、个人博客
   - 投入产出比低，适合碎片时间测试

4. **放弃/暂缓**（左下）：
   - 攻击面小 + WAF强
   - 大厂核心业务（有安全团队）
   - 新手别碰，老手可以挑战
