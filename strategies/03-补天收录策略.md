## 1. 账号准备

### 1.1 注册补天账号

**注册流程**：

```
Step 1: 访问补天官网 https://butian.360.cn
Step 2: 点击右上角「注册」
Step 3: 填写手机号、邮箱、用户名
Step 4: 实名认证（必须）
Step 5: 完善个人资料
Step 6: 等待审核通过（通常1-2个工作日）
```

**注册注意事项**：

1. **手机号选择**：
   - 建议使用长期使用的手机号注册（审核可能需要验证码）
   - 不要使用一次性号码或虚拟号码
   - 注册后不要更换手机号，否则需要重新认证

2. **邮箱选择**：
   - 推荐使用 QQ邮箱 或 163邮箱（Gmail可能收不到某些通知）
   - 邮箱应为常用邮箱，用于收漏洞通知
   - 建议绑定邮箱并开启通知

3. **用户名选择**：
   - 选择一个专业的用户名（不要用奇怪的名字）
   - 用户名在补天平台是公开的，会影响你的「江湖地位」
   - 建议与各大SRC平台保持同一用户名（建立个人品牌）
   - 示例：`sec_xxx` `whitehat_xxx` `xxx_sec`

4. **实名认证（关键）**：
   - 补天要求所有白帽子必须实名认证
   - 需要上传身份证照片（正面+反面）
   - 需通过人脸识别验证
   - 实名认证信息严格保密，不会对外泄露
   - 没有实名认证无法提现奖金！

### 1.3 了解补天平台规则

**必须阅读的文档**：

在开始参与补天之前，务必仔细阅读以下规则：

1. **《补天漏洞审核标准》**：了解什么漏洞收、什么不收
2. **《补天用户协议》**：了解你的权利和义务
3. **《补天奖励规则》**：了解如何计算积分和奖金
4. **《补天目标范围说明》**：了解如何确认测试范围
5. **《补天免责声明》**：了解法律风险

**重要规则摘要**：

```
1. 漏洞提交期限：发现漏洞后72小时内提交（超时可能不被收录）
2. 禁止重复提交：同一漏洞只收录第一个提交者
3. 禁止恶意破坏：拖库、删数据、修改配置等行为将被封号
4. 禁止私下交易：不得与企业私下交易漏洞
5. 禁止公开漏洞：未修复的漏洞不能公开讨论
6. 禁止多账号注册：一个自然人只能注册一个账号
7. 数据脱敏要求：提交漏洞时必须对数据进行脱敏处理
8. 测试行为约束：测试过程中不能影响正常业务
```

## 2. 目标认领

### 2.1 如何正确认领目标

**目标认领流程**：

```
Step 1: 浏览补天目标列表
Step 2: 找到感兴趣的目标，点击「查看详情」
Step 3: 确认目标测试范围和授权状态
Step 4: 点击「认领目标」
Step 5: 确认已阅读并同意测试协议
Step 6: 开始测试
```

**认领注意事项**：

1. **确认授权状态**：
   - 目标必须显示「已授权」才能测试
   - 没有授权确认的目标不要测试
   - 不确定的可以通过补天客服确认

2. **确认测试范围**：
   - 仔细阅读目标的「测试范围」说明
   - 确认哪些域名/IP在范围内
   - 有些目标有「限制测试」的说明（如限制扫描器、限制时间等）

3. **目标规则**：
   - 有些目标有「禁测事项」，比如不能测某类漏洞
   - 有些目标有「优先测试方向」，按需求测更可能被收录
   - 目标规则的更新：定期检查已认领目标的规则变化

### 2.2 目标认领策略

**什么样的目标值得认领**：

```
优先级1（立即认领）：
- 新上线的目标（其他白帽子还没开始挖）
- 与你的技术专长匹配的目标
- 赏金高且测试范围大的目标
- 你之前对这个行业/技术栈熟悉的目标

优先级2（可以考虑）：
- 其他白帽子已经提交了漏洞但还有机会的
- 测试范围适中但赏金还可以的
- 你不太熟悉但想学习的目标

优先级3（不要认领）：
- 已经有很多人认领的目标（竞争激烈）
- 测试范围极小的目标（攻击面太小）
- 赏金太低的目标（投入产出比低）
- 明确标注「不活跃」的目标
```

**认领数量管理**：

```markdown
初学者：
- 同时认领 1-2 个目标（集中精力，避免分散）
- 等一个目标有产出后再认领下一个

中级白帽子：
- 同时认领 3-5 个目标（主攻1个，副攻2-3个，Recon 1-2个）
- 目标类型多样化（简单+中等各半）

高级白帽子：
- 同时认领 5-10 个目标（根据时间安排）
- 目标类型覆盖简单-困难
- 部分目标进行持续监控（新功能上线=新漏洞机会）
```

### 2.3 目标授权范围确认

**授权范围确认方法**：

```bash
# 方法1：查看补天目标详情页
# 确认列出的「测试范围」包含哪些域名/IP

# 方法2：子域名排查
# 使用 subfinder 发现目标的子域名
subfinder -d example.com -all -o subdomains.txt

# 然后对比目标范围
# 如果发现某个子域名不在范围内，可以先不测
# 或者向补天申请扩大范围

# 方法3：IP归属确认
# 确认IP是否确实属于目标公司
whois 1.2.3.4
# 或者使用 ipinfo.io
curl https://ipinfo.io/1.2.3.4
```

**授权范围常见问题**：

```
Q: 目标只写了 example.com，我可以测试 sub.example.com 吗？
A: 通常可以，子域名属于同一父域名的资产。
  但最好先确认目标范围说明中有无限制。

Q: 目标写了 www.example.com，我可以测试 admin.example.com 吗？
A: 同上，通常子域名都在范围内。
  但更稳妥的做法是：先报告给补天客服确认。

Q: 目标只给了 IP 范围，我可以测试该IP的所有端口吗？
A: 可以，但注意不要影响目标业务。
  端口扫描时使用低速率，避免对业务造成影响。

Q: 发现目标使用了 CDN，CDN IP 在范围内吗？
A: CDN IP 通常不在范围内，需要测试真实源IP。

Q: 发现目标有测试环境 test.example.com，可以测试吗？
A: 取决于目标范围说明。通常测试环境也在范围内，
  而且测试环境的安全性通常更差。
```

## 3. 漏洞提交技巧

### 3.1 标题怎么写才吸引审核

**标题的核心原则**：**简洁、准确、有冲击力**

标题是审核人员第一眼看到的内容。一个好的标题决定了审核人员对漏洞的第一印象。

**标题结构**：

```
[漏洞类型] + [漏洞位置] + [危害说明]

示例：
「SQL注入漏洞」+「在商品搜索接口」+「可获取全量用户数据」
→ SQL注入漏洞 - 商品搜索接口存在报错注入，可获取全量用户数据
```

**标题模板**：

```
高危漏洞标题模板：
- SQL注入漏洞 - [接口名] 存在联合查询注入，可获取数据库全量数据
- 越权漏洞 - [接口名] 未校验用户身份，可查看任意用户[敏感信息]
- 文件上传漏洞 - [功能点] 存在任意文件上传漏洞，可获取服务器权限
- 命令执行漏洞 - [功能点] 存在远程命令执行漏洞，可执行任意系统命令
- SSRF漏洞 - [功能点] 存在SSRF漏洞，可访问内网[服务名]

中危漏洞标题模板：
- 存储XSS - [功能点] 存在存储型XSS漏洞，可对用户实施钓鱼攻击
- 越权漏洞 - [功能点] 存在水平越权，可查看其他用户的[非敏感数据]
- 逻辑漏洞 - [功能点] 存在[具体逻辑问题]，可[造成什么影响]
- 信息泄露 - [接口名] 泄露用户[数据类型]，涉及大量用户

低危漏洞标题模板：
- 反射XSS - [功能点] 存在反射型XSS漏洞（无HttpOnly Cookie）
- URL跳转 - [功能点] 存在URL跳转漏洞，可用于钓鱼攻击
- 弱口令 - [系统名] 存在弱口令漏洞 `admin/123456` 可登录后台
```

**标题示例（好的 vs 坏的）**：

```markdown
❌ 坏的标题：
- "网站有个漏洞"
- "SQL注入"
- "一个越权漏洞"
- "信息泄露"
- "发现一个问题"

✅ 好的标题：
- "SQL注入漏洞 - 用户登录接口存在时间盲注，可枚举数据库表名"
- "越权漏洞 - 订单查询接口未校验用户身份，可查看任意用户订单详情（含收货地址）"
- "存储XSS - 个人资料编辑功能存在存储型XSS，可获取管理员Cookie"
- "逻辑漏洞 - 优惠券功能存在刷单逻辑漏洞，可无限领取优惠券"
- "信息泄露 - Swagger API文档未关闭，泄露所有API接口及参数详情"
```

### 3.2 影响描述怎么写才被认定为高危

**影响描述的核心原则**：**说清楚这个漏洞能造成什么实际危害**

很多白帽子只写了「存在SQL注入」，但没有说清楚注入后能干什么。好的影响描述应该让审核人员一看就知道：这个漏洞很严重。

**影响描述结构**：

```
1. 攻击者能够做什么（具体）
2. 能够获取到什么数据（具体）
3. 对业务的影响（具体）
4. 利用难度（证明很容易利用）
```

**不同漏洞类型的影响描述模板**：

**SQL注入**：

```
【影响描述】
1. 攻击者可以通过该SQL注入漏洞，执行任意SQL语句，包括：
   - 查询数据库表名、字段名、数据
   - 读取敏感数据（用户信息、订单信息、管理密码等）
   - 写入webshell（如果有权限）
2. 该数据库包含XX万用户信息，包括：手机号、身份证号、家庭住址等敏感信息
3. 该漏洞无需特殊权限（任何用户/无需登录即可触发）
4. 该漏洞可稳定利用，利用难度低

【证明】
- 成功查询到数据库中用户表的数据：XXX（已脱敏）
- 数据库用户为root（有写入权限）
```

**越权漏洞**：

```
【影响描述】
1. 攻击者可以越权查看/操作其他用户的：
   - 个人资料（姓名、电话、地址）
   - 订单信息（商品、金额、物流）
   - 交易记录（充值、消费、余额）
2. 该漏洞影响平台所有用户（XX万+）
3. 无需用户交互，直接通过API请求即可触发
4. 参数可遍历，可以批量获取所有用户的数据

【证明】
- 用户A成功查看用户B的完整资料（附截图，已脱敏）
- 遍历ID从1-1000，所有用户信息均可获取
```

**RCE/命令执行**：

```
【影响描述】
1. 攻击者可以在服务器上执行任意系统命令
2. 可以控制整台服务器，包括：
   - 读取服务器上的所有文件
   - 获取数据库连接信息
   - 植入后门/挖矿程序
   - 横向移动到内网其他服务器
3. 该漏洞无需特殊权限即可触发
4. 服务器配置为Web服务器，对外提供核心业务服务

【证明】
- 执行命令 whoami 返回 www-data（附截图）
- 执行命令 cat /etc/passwd 确认服务器信息（已脱敏）
- 确认 outbound 网络可访问外网（可反弹Shell）
```

### 3.3 复现步骤怎么写才清晰

**复现步骤的核心原则**：**一个从零开始的、可复现的、完整步骤**

审核人员会在测试环境按照你的步骤尝试复现漏洞。如果步骤不清晰，审核人员无法复现，漏洞可能被驳回。

**复现步骤结构**：

```
【环境准备】
（需要的工具、账号、前提条件等）

【步骤一】做什么
（具体的操作，包括URL、参数、请求体等）

【步骤二】看到什么
（期望的观察结果，附截图）

【步骤三】等等
```

**完整示例**：

```
【漏洞名称】SQL注入 - 商品搜索接口存在报错注入

【漏洞URL】https://target.com/api/product/search

【漏洞参数】keyword

【请求方式】GET

【复现步骤】

Step 1: 登录平台，获取有效的Session Token
  - 使用测试账号 test@example.com / test123456 登录
  - 从响应中获取 session_token

Step 2: 构造恶意请求
  - 正常请求：
    GET /api/product/search?keyword=手机 HTTP/1.1
    Host: target.com
    Authorization: Bearer [session_token]
  
  - 注入测试请求：
    GET /api/product/search?keyword=手机' HTTP/1.1
    Host: target.com
    Authorization: Bearer [session_token]

Step 3: 观察响应
  - 正常请求返回：{"code":200, "data": [...]}
  - 注入测试返回：{"code":500, "msg": "You have an error in your SQL syntax..."}
  确认存在SQL注入

Step 4: 提取数据库信息（证明危害）
请求：
GET /api/product/search?keyword=手机' AND extractvalue(1, concat(0x7e, (SELECT database()))) --
响应：
{"code":500, "msg": "XPATH syntax error: '~ecommerce_db'"}
确认数据库名为 ecommerce_db

Step 5: 提取用户数据（证明高危）
请求：
GET /api/product/search?keyword=手机' AND extractvalue(1, concat(0x7e, (SELECT group_concat(username) FROM users))) --
响应：
{"code":500, "msg": "XPATH syntax error: '~admin,user1,user2,...'"}
确认可读取用户表中的数据

【截图附件】
- 图1：正常请求与响应的对比
- 图2：SQL报错的响应
- 图3：提取数据库名
- 图4：提取用户表数据（已对敏感信息打码）
- 图5：Burp Suite抓包截图（包含完整的请求包）

【影响证明】
- 成功确认 SQL注入点
- 成功获取数据库版本、用户、表名
- 成功提取用户表数据（已脱敏）
- 数据库用户为 root，运行在Linux服务器上
```

### 3.4 截图/PoC怎么附

**截图要求**：

```markdown
截图黄金法则：

1. 清晰度
   - 截图必须清晰，文字可读
   - 不要用手机拍屏幕照片
   - 推荐使用截图工具：Snipaste / QQ截图 / Flameshot

2. 关键信息
   - 包含URL（地址栏或Burp的请求包Host）
   - 包含请求包和响应包（Burp的Raw标签页）
   - 包含关键数据（已打码敏感信息）
   - 包含时间（证明时效性）

3. 不要包含
   - 不相关的页面元素（工具栏、书签、其他标签页）
   - 真实用户数据（必须打码/模糊化）
   - 过多的重复截图

4. 截图整理
   - 按步骤编号命名（01-step1.png, 02-step2.png）
   - 每张截图要有标题说明
   - 使用截图画框/箭头标注关键位置
   - 合并为PDF或ZIP打包提交
```

**Burp Suite截图标准**：

```
截图1：Proxy → HTTP History
- 显示完整的请求URL
- 显示请求方法和参数
```

**PoC（Proof of Concept）怎么写**：

```python
# PoC Python脚本模板
# 用于证明漏洞存在的概念验证代码

import requests
import sys

def poc(target_url, session_token):
    """验证SQL注入漏洞的PoC"""
    
    headers = {
        "Authorization": f"Bearer {session_token}",
        "User-Agent": "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36"
    }
    
    # Step 1: 正常请求
    normal_params = {"keyword": "手机"}
    r1 = requests.get(
        f"{target_url}/api/product/search",
        params=normal_params,
        headers=headers
    )
    print(f"[+] 正常请求状态码: {r1.status_code}")
    print(f"[+] 正常请求返回: {r1.text[:100]}...")
    
    # Step 2: SQL注入测试
    inject_params = {"keyword": "手机' AND 1=1 -- "}
    r2 = requests.get(
        f"{target_url}/api/product/search",
        params=inject_params,
        headers=headers
    )
    print(f"[+] 注入(AND 1=1)状态码: {r2.status_code}")
    print(f"[+] 注入(AND 1=1)返回: {r2.text[:100]}...")
    
    inject_params2 = {"keyword": "手机' AND 1=2 -- "}
    r3 = requests.get(
        f"{target_url}/api/product/search",
        params=inject_params2,
        headers=headers
    )
    print(f"[+] 注入(AND 1=2)状态码: {r3.status_code}")
    print(f"[+] 注入(AND 1=2)返回: {r3.text[:100]}...")
    
    if r2.status_code == 200 and r3.status_code in [500, 403, 400]:
        print("[!] 确认存在SQL注入！")
        return True
    else:
        print("[-] 未确认SQL注入")
        return False

if __name__ == "__main__":
    print("=" * 50)
    print("SQL注入漏洞PoC")
    print("=" * 50)
    
    target = sys.argv[1] if len(sys.argv) > 1 else "https://target.com"
    token = sys.argv[2] if len(sys.argv) > 2 else ""
    
    poc(target, token)
```

**PoC脚本编写要点**：

```
1. 简单明了：不要写复杂的脚本，只需要证明漏洞存在
2. 自动化：最好是一个可以一键运行的脚本
3. 注释清晰：每一步做了什么，为什么
4. 错误处理：友好提示
5. 脱敏处理：不要在PoC中包含真实密码/Token（用占位符代替）
6. 安全提示：在PoC开头加上免责声明
```

## 4. 审核周期

### 4.1 提交后多久出结果

**补天审核时间线**：

```
提交漏洞
    │
    ▼
┌─────────────────┐
│  初审（1-3个工作日） │
│  审核人员初步判断漏洞 │
│  是否有效、是否重复   │
└────────┬────────┘
         │
         ▼
  ┌──────────────┐
  │  初审通过     │
  │  → 进入复审    │
  └──────────────┘
         │
         ▼
┌─────────────────┐
│  复审（3-7个工作日） │
│  企业确认漏洞是否存在  │
│  企业评估漏洞危害等级  │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌──────┐   ┌──────┐
│ 通过  │   │ 驳回  │
│ →收录 │   │ →申诉 │
└──────┘   └──────┘
```

**不同漏洞类型的审核时间**：

| 漏洞类型 | 初审时间 | 复审时间 | 总时间 |
|---------|---------|---------|-------|
| 高危（SQL注入/RCE等） | 1-2天 | 3-5天 | 4-7天 |
| 中危（XSS/越权等） | 1-3天 | 5-7天 | 6-10天 |
| 低危（反射XSS等） | 2-5天 | 5-10天 | 7-15天 |
| 需要确认的 | 3-7天 | 7-14天 | 10-21天 |

**影响审核速度的因素**：

```
1. 漏洞清晰度
   - 复现步骤清晰 → 审核快
   - 复现步骤模糊 → 审核慢或驳回

2. 漏洞类型
   - 高危漏洞 → 审核更快（企业重视）
   - 低危漏洞 → 审核慢（优先级低）

3. 审核人员繁忙程度
   - 工作日 → 审核快
   - 节假日 → 审核慢（假期积压）

4. 企业响应速度
   - 活跃目标 → 复审快
   - 不活跃目标 → 复审慢

5. 是否为重复漏洞
   - 新漏洞 → 正常流程走完
   - 重复漏洞 → 2-3天直接驳回
```

### 4.2 审核状态说明

| 状态 | 含义 | 预计时间 |
|------|------|---------|
| 待审核 | 已提交，等待审核人员初审 | 1-3天 |
| 审核中 | 审核正在进行 | 3-7天 |
| 待确认 | 审核人员需要你补充信息 | 尽快回复 |
| 已通过 | 漏洞已被收录 | - |
| 已驳回 | 漏洞不被收录（有原因说明） | - |
| 重复 | 该漏洞已有人提交 | 立即 |
| 挂起 | 企业需要更多时间评估 | 14天+ |

### 4.3 如何跟进审核进度

**不要频繁催促**：

```
错误做法：
  - 每天发消息问审核进度（会被烦死）
  - 一天内提交多个版本的漏洞（反而会乱）
  - 威胁审核人员（会被拉黑）

正确做法：
  - 提交后等一周，如果还没消息再询问
  - 通过补天官方客服咨询，不要直接联系审核人员
  - 在漏洞补充说明区添加补充信息
```

**审核中被要求补充信息怎么办**：

```
1. 及时响应
   - 审核人员要求补充信息时，尽快回复
   - 通常审核人员会说明需要补充什么
   - 按要求补充，不要多也不要少

2. 常见补充要求
   - "请提供更详细的请求包截图"
   - "请确认该漏洞是否可稳定复现"
   - "请补充漏洞对业务的实际影响"
   - "请提供更多的测试数据证明"

3. 补充信息格式
   - 按审核人员要求的格式补充
   - 如需补充截图，在原漏洞附件中上传
   - 如需补充文字，在评论区回复
```

## 5. 申诉技巧

### 5.1 被打回后怎么办

**被打回的常见原因**：

```
1. 漏洞已存在（重复提交）
   - 有人在您之前提交了相同漏洞
   - 无法申诉，只能接受

2. 无法复现
   - 审核人员按照你的步骤无法复现漏洞
   - 需要提供更详细的复现步骤或截图

3. 影响不够
   - 审核人员认为漏洞危害不足
   - 需要补充漏洞的实际影响证明

4. 不在范围内
   - 漏洞涉及的资产不在目标范围内
   - 无法申诉，检查目标范围

5. 属于不收录类型
   - 该漏洞类型在补天的不收录列表中
   - 无法申诉

6. 信息不足
   - 提交的漏洞信息不足以判断
   - 补充信息后重新提交

7. 误判
   - 审核人员误判了漏洞类型或危害程度
   - 可以进行申诉（提供证据）
```

### 5.2 申诉流程

**申诉流程详解**：

```
Step 1: 阅读驳回原因
  仔细阅读审核人员给出的驳回原因
  理解为什么被驳回

Step 2: 评估是否有申诉价值
  如果驳回原因是：
  ✅ 无法复现 → 可以申诉（补充详细信息）
  ✅ 影响不够 → 可以申诉（补充危害证明）
  ✅ 误判漏洞类型 → 可以申诉（提供证据）
  ❌ 重复漏洞 → 不要申诉
  ❌ 不在范围内 → 不要申诉
  ❌ 不收录类型 → 不要申诉

Step 3: 准备申诉材料
  根据驳回原因准备对应的补充材料

Step 4: 提交申诉
  在漏洞页面点击「申诉」按钮
  填写申诉理由并上传补充材料

Step 5: 等待复审
  申诉提交后进入复审流程
  通常3-7个工作日出结果
```

## 6. 积分与排名

### 6.1 补天积分体系

**积分规则概述**：

```
积分 = 漏洞基础分 + 加分

漏洞基础分：
  严重漏洞: 100-150 分
  高危漏洞: 50-99 分
  中危漏洞: 20-49 分
  低危漏洞: 5-19 分

加分项：
  首次提交漏洞: +5 分
  高质量报告: +10 分（清晰的复现步骤+完整截图）
  高危+严重: +15 分（额外奖励）
  连续提交: +5 分（连续30天有提交）
  企业好评: +20 分（企业特别表扬）
  受邀测试: +30 分（企业邀请测试）
```

### 6.2 如何快速提升积分

**积分提升策略**：

```
策略1：集中精力挖高危
  1个高危（50-99分）≈ 2-3个中危（20-49分）≈ 5-10个低危（5-19分）
  结论：把精力放在高危漏洞上

策略2：追求高质量报告
  每次提交都做到极致：
  - 清晰的标题
  - 完整的步骤
  - 清晰的截图
  - 有效的PoC
  高质量报告可能获得加分

策略3：抓住首发漏洞
  新目标上线时立即认领
  新功能上线时立即测试
  首发漏洞没有竞争

策略4：深耕一个目标
  了解一个目标的所有业务
  了解一个目标的代码风格
  了解一个目标的安全短板
  在一个目标上找到多个漏洞

策略5：持续提交
  连续提交有额外加分
  不要挖到一个就停
  保持稳定的提交节奏
```

**积分与排名的关系**：

```
排名段    积分门槛    特征
Top 10    10000+     大神级，全职挖洞
Top 50    3000-9999  高级白帽子，月入过万
Top 100   1000-2999  中级白帽子，稳定产出
Top 500   200-999    初级白帽子，持续提交
新手区    0-199      刚入门的学习者
```

### 6.3 排名带来的好处

**高排名的优势**：

```
1. 企业邀请
   - 高排名白帽子会被企业主动邀请测试
   - 受邀测试通常有额外奖金
   - 受邀测试的目标通常质量更高

2. 审核优先级
   - 高排名白帽子的漏洞审核优先
   - 审核人员更重视高排名白帽子的提交

3. 漏洞溢价
   - 有些企业会给高排名白帽子额外奖励
   - 信任度高，企业更愿意采纳建议

4. 行业认可
   - 简历亮点（求职/跳槽有优势）
   - 安全圈人脉

5. 奖金提现
   - 高排名提现优先级更高
   - 提现速度更快
```

## 7. 合规红线

### 7.1 绝对不能做的事

**行为红线**（一旦触犯，直接封号，可能承担法律责任）：

```markdown
═══════════════════════════════════════════
              行为红线
═══════════════════════════════════════════

【1】拖库/下载数据
   ❌ 不要从数据库中下载大量数据
   ❌ 不要导出用户表格
   ❌ 不要保存敏感数据到本地
   ✅ 只做证明：用 count(*) 证明数据存在
   ✅ 只获取几条数据作为证据（必须脱敏）

【2】删除/修改数据
   ❌ 不要执行 DELETE/UPDATE/DROP 语句
   ❌ 不要删除用户数据
   ❌ 不要修改系统配置
   ✅ 只做 SELECT 查询
   ✅ 需要证明写入能力时，先申请再操作

【3】破坏服务
   ❌ 不要进行DoS/DDoS攻击
   ❌ 不要执行大负载查询（sleep长时间）
   ❌ 不要大量并发请求
   ✅ 控制测试力度，不影响正常业务

【4】未授权渗透
   ❌ 不要测试未授权的目标
   ❌ 不要超出授权范围
   ❌ 不要测试不在范围内的第三方服务
   ✅ 严格按照补天授权的目标范围

【5】利用漏洞谋利
   ❌ 不要利用漏洞获取私利
   ❌ 不要利用漏洞提现/转账
   ❌ 不要利用漏洞获取VIP权限
   ✅ 通过补天平台合法赚取赏金

【6】私下交易
```
